# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 프로젝트 목적

Karpathy의 "LLM Wiki" gist (https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) 를 팀에 소개하기 위한 **5장짜리 발표자료**를 제작한다. 코드베이스가 아니라 슬라이드 자료 저장소다.

- **청중:** 본부 임원진 (본부장 참석). 개발 조직 출신이지만 LLM 실무자는 아님.
- **자리의 성격:** 공식·중요. "흥미로운 기술 소개"가 아니라 **결정거리를 드리는 자리**로 마무리되어야 한다.
- **결과물 길이:** 정확히 5장. 늘리지 말 것.

## 작업 시 핵심 원칙

1. **"왜 우리 조직이 지금 이걸 봐야 하는가" 관점을 유지한다.** 기술 설명에 매몰되지 말 것. 매 슬라이드를 임원의 의사결정 흐름에 맞춰라.
2. **권위를 빌린다.** Karpathy를 모르는 임원이 있을 수 있으므로 1슬라이드에서 "전 Tesla AI 디렉터, OpenAI 창립 멤버" 한 줄을 항상 유지.
3. **추상 → 구체로 내려온다.** 슬라이드 4는 가장 중요한 슬라이드이며, 본부의 실제 문서 흐름(분기 보고, 고객 미팅 노트, 외부 리서치 등)에 매핑된 시나리오가 들어가야 한다. 일반론 금지.
4. **마지막 슬라이드는 반드시 결정 요청으로 끝난다.** 파일럿 범위 / 담당자 / 보고 시점 — 이 셋이 빠지면 자료 가치가 절반.
5. **핵심 인용은 원문에서 그대로 가져와 강조한다.**
   - "Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다."
   - "위키가 실패해온 이유는 유지보수 비용이었고, LLM은 그 비용을 0에 수렴시킨다."

## 5장 구성 (확정)

| # | 역할 | 핵심 메시지 |
|---|------|-------------|
| 1 | Hook | 한 줄 요약 + Karpathy 권위 부여 |
| 2 | 문제 정의 | "현재 RAG/NotebookLM은 매번 지식을 재발견한다 — 축적되지 않는다" + 비교 시각화 (RAG vs Wiki) |
| 3 | 핵심 아이디어 | 3-layer 다이어그램: Raw sources → Wiki → Schema(CLAUDE.md/AGENTS.md) |
| 4 | 조직 적용 | 본부 실제 문서 흐름 매핑 + 3 운영 사이클(Ingest / Query / Lint) |
| 5 | 시사점 & 제안 | 왜 지금인지 + 4~6주 파일럿 제안 + 결정 요청 3가지 |

슬라이드 4가 가장 무겁고, 1·5가 그 다음. 2·3은 짧고 명료하게.

## 작업 파일

- `초안.md` — 5장 구성의 원본 초안. 톤·메시지의 근거. 수정 시 이 파일과 일관성을 유지하거나 변경 이유를 명시한다.
- 슬라이드 도구(Marp/Reveal/PPTX 등)는 아직 정해지지 않음. 파일을 만들기 전에 사용자에게 확인할 것.

## 톤 가이드

- 한국어, 임원 보고 톤 (격식 있되 길지 않게).
- 문장은 짧게. 슬라이드 본문에는 한 슬라이드당 핵심 메시지 1개 + 보조 3줄 이내.
- "흥미롭다" / "혁신적이다" 같은 평가어 대신 구체적 효과·숫자·시나리오로 말한다.
- 리스크가 작아 보이도록 표현한다 — 임원이 OK 하기 쉬워야 한다 ("본부 전체"가 아닌 "작은 영역부터").
