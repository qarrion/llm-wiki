## 슬라이드 2 — 문제 정의: 지금 우리가 LLM을 쓰는 방식의 한계

**핵심 메시지**
> "현재의 RAG / NotebookLM / ChatGPT 파일 업로드는 매번 처음부터 지식을 재발견한다."

**강조 포인트 (1개로 압축)**
- **지식이 축적되지 않는다.** 같은 질문을 두 번 하면 LLM은 두 번 다 흩어진 조각을 처음부터 다시 모은다. 지난 분석은 채팅 로그 어딘가에 사라진다.

**시각화 — 좌우 비교 (1장에 들어가야 함)**

| 현재 방식 (RAG / NotebookLM) | LLM Wiki |
|---|---|
| 매 질문마다 raw 문서 재검색·재조립 | 한 번 컴파일된 지식이 축적·유지 |
| 답변은 채팅 로그로 휘발 | 좋은 답변은 위키에 다시 적립 (closed loop) |
| 5문서 종합 질문 → 매번 처음부터 | 교차 참조·모순 표시가 이미 되어 있음 |
| 시간이 지나도 그대로 | 시간이 지날수록 풍부해짐 |

**(v2 추가) 원문 인용 (작은 글씨로 한 줄)**
> "RAG는 매번 처음부터 지식을 재발견한다. 어떤 것도 축적되지 않는다." — Karpathy

## 슬라이드 3 — 핵심 아이디어: LLM Wiki란 무엇인가

**3-Layer 구조 (다이어그램으로)**

```
Raw sources       →    Wiki                    →    Schema
(원본 문서, 불변)       (LLM이 작성·유지하는           (CLAUDE.md / AGENTS.md
 LLM은 읽기만)          마크다운 페이지들)              위키 운영 규칙)
```

**핵심 비유 (강조 인용 — 슬라이드 정중앙)**
> **"Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다."**

**(v2 추가) 두 번째 비유 (개발 조직 출신 임원에게 더 직관적)**
> 그리고 한 단계 더:
> **"LLM은 지식 컴파일러다.** raw 문서를 입력으로 받아 구조화된 wiki(.md 그래프)를 출력한다."

> 💡 두 비유 다 개발 조직 출신 임원에게 즉시 꽂힌다. 둘 중 하나만 써도 되지만, **임원진 구성에 따라 두 번째 비유가 더 강하게 박힐 수 있음.**

**(v2 추가) RAG 인프라가 필요 없다는 단순함 (작은 박스)**
- `index.md` (전체 페이지 카탈로그) + `log.md` (시간순 작업 로그) — 이 두 파일만으로도 ~100개 문서 / ~40만 단어 규모까지 잘 작동.
- 즉 시작에 별도 벡터 DB·임베딩 파이프라인 불필요. **첫 6주에 인프라 비용은 사실상 0.**

## 슬라이드 4 — 우리 조직에 어떻게 적용 가능한가 ⭐ (가장 중요한 슬라이드)

> 본부장님이 가장 집중해서 듣는 슬라이드. **추상 → 본부의 구체 시나리오로 반드시 내려와야 한다.** 일반론 금지.

**원문이 명시한 비즈니스 적용 (그대로 인용)**
> "Slack 스레드, 회의록, 프로젝트 문서, 고객 통화를 LLM이 흡수해 유지하는 내부 위키. 필요하면 사람이 검토 단계에 참여한다." — Karpathy, Gist

**(v2 핵심 보강) 본부 매핑 — 발표 전까지 1~2개 골라 채워넣을 자리**

> 아래 4개 후보 중 **본부 실제 문서 흐름 1~2개를 선택**해 *"이 흐름이 이렇게 흘러갈 수 있다"* 로 구체화.
> ※ 이 자리가 비면 슬라이드 4의 가치는 절반 이하.

| 후보 | 현재 문제 (예상) | LLM Wiki 적용 시 |
|---|---|---|
| 분기 보고 | 매번 흩어진 자료 재취합 | 위키가 자동 누적, 분기마다 diff만 추가 |
| 고객 미팅 노트 | 노트 어디 있는지 못 찾음 | 고객별 엔티티 페이지 자동 유지 |
| 외부 리서치 | 한 번 읽고 사라짐 | 리서치마다 위키 풍부해짐 |
| (본부 고유 흐름) | — | — |

**3가지 운영 사이클 (작은 박스 3개로)**

| 사이클 | 내용 | (v2) 본부 관점 한 줄 |
|---|---|---|
| **Ingest** | 새 문서를 위키에 흡수 (1소스 → 10~15페이지 연쇄 갱신) | "한 번 넣으면 관련 페이지 다 알아서 갱신됨" |
| **Query** | 질문 → 답변, **그리고 그 답변도 다시 위키에 적립** | "조사한 게 채팅 로그로 사라지지 않음" (closed loop) |
| **Lint** | 주기적 일관성 점검 · 모순 탐지 · 누락 보완 | "위키가 시간이 지날수록 깨끗해짐" |

**(v2 추가) Closed loop 한 줄 강조**
> 이 셋 중 본부에 가장 강력한 건 **Query의 closed loop**.
> "임원·팀원이 LLM에게 던진 질문 → 받은 답변이 본부 지식 자산으로 자동 누적." 즉 *질문하는 행위 자체가 자산을 만든다.*

## 슬라이드 5 — 시사점 & 제안 (Action Items)

> "흥미로운 자료였습니다"로 끝나면 자료 가치는 절반.
> **의사결정자에게는 항상 결정거리를 드린다.**

### ① 왜 지금이 적기인가 (3 신호)

1. **(인용)** "위키가 실패해온 이유는 유지보수 비용이었고, LLM은 그 비용을 0에 수렴시킨다." — Karpathy
2. **(v2 추가) 역사적 맥락:** 1945년 Vannevar Bush가 제안한 **Memex** — 개인 지식 시스템의 80년 비전. 여태 막혀 있던 단 하나의 병목이 "유지보수 주체"였고, **그 병목을 LLM이 처음으로 풀었다.**
3. **(v2 추가) 시장 신호:** Karpathy 본인이 *"단순 스크립트 묶음이 아니라 제품이 될 수 있다"* 고 명시. 비전과 실행 스펙이 동시에 공개된 시점 = **선점 가능 구간이 아직 짧게 남아 있다는 신호.**

### ② 시범 적용 제안

- **범위:** 본부 전체 ❌ → **작은 영역 하나** ✅
  - 예: 특정 프로젝트 문서 묶음 1개, 또는 한 팀의 외부 리서치 자료
- **기간:** 4~6주 파일럿
- **비용:** **인프라 비용 사실상 0** (벡터 DB·RAG 파이프라인 불필요. `index.md` + `log.md` + LLM 에이전트만으로 시작)
- **포지셔닝:** 리스크가 작아 보여야 임원이 OK 하기 쉬움. *"실패해도 사라지는 건 한 폴더의 마크다운 파일뿐."*

### ③ 결정 요청 사항 (이 자리에서 받아야 할 답)

- [ ] **파일럿 범위** — 어떤 영역? (위 후보 중 1개)
- [ ] **담당자** — 누가 운영? (1명, 부담 낮음)
- [ ] **보고 시점** — 4주 차 / 6주 차 중 언제 결과 보고?

> ※ 예산 항목은 의도적으로 빼거나, *"별도 예산 없음"* 으로 한 줄만 남긴다. 예산 협상은 이 자리의 목적이 아니다.
