### 권장: **B를 헤드라인, A를 보조로.** D는 발표자 멘트로 흡수.

> **헤드라인:** 같은 질문을 두 번 하면, LLM은 두 번 다 처음부터 시작한다.
> **보조 한 줄:** RAG·NotebookLM·ChatGPT 파일 업로드 — 모두 같은 한계.

이유:
- B는 임원 본인의 *지난주 경험*에 즉시 닿는다 ("작년 분기 보고 자료를 다시 묻다가 답이 매번 달라서 짜증 났던 그 순간").
- A는 *원문 권위* 를 깔아주는 보조. B만 있으면 카피처럼 보일 위험이 있다.
- D ("매번 새로운 인턴")는 발표 도중 임원과 눈을 맞추며 던지는 멘트로 가장 강하다. 슬라이드에 박아두면 카피 톤이 됨 → 구두로만 쓴다.

## 2. 시각 구성 옵션

### 옵션 ㄱ — 좌우 비교표 (ver2 본문 안)

```
[현재 방식 (RAG)]            [LLM Wiki]
매 질문마다 raw 재검색·재조립  →  한 번 컴파일된 지식이 축적
답변은 채팅 로그로 휘발       →  좋은 답변은 위키에 다시 적립
5문서 종합 → 매번 처음부터    →  교차 참조·모순 표시가 이미 되어 있음
시간이 지나도 그대로          →  시간이 지날수록 풍부해짐
```

- **장점:** 한눈에 비교. 표 형식이 임원 보고 톤에 맞음.
- **단점:** *슬라이드 3의 답을 슬라이드 2에서 미리 보여주는* 구조 — 슬라이드 3의 임팩트를 약화시킴.

### 옵션 ㄴ — "현재 방식만" 다이어그램 (질문 → RAG → 답 → 휘발)

```
[질문 1] → [raw 100문서 재검색] → [답변] → ⨯ 휘발
[질문 2] → [raw 100문서 재검색] → [답변] → ⨯ 휘발  ← 똑같은 비용 다시
[질문 3] → [raw 100문서 재검색] → [답변] → ⨯ 휘발  ← 똑같은 비용 또
```

- **장점:** *문제만* 보여주고 답은 슬라이드 3으로 미룬다. 슬라이드 간 긴장감 유지.
- **단점:** 그림 그리는 부담.

### 옵션 ㄷ — 본부 시나리오 1개 (구체)

> 예시: *"분기 보고를 위해 LLM에 작년 자료 100건을 올려 질문했다. 다음 분기에 같은 질문을 했을 때, LLM은 100건을 다시 처음부터 읽었다. 그동안 정리해둔 분석은 어디에도 남지 않았다."*

- **장점:** 가장 임원에 닿는다. 슬라이드 4보다 먼저 *"우리 일"* 신호를 한 번 던지는 효과.
- **단점:** 슬라이드 4와 본부 시나리오가 겹쳐 메시지 중복 위험. **슬라이드 4용 시나리오와 다른 것을 골라야 한다.**

### 권장: **옵션 ㄴ 메인 + 옵션 ㄷ 1줄 캡션.** 옵션 ㄱ은 슬라이드 3로 옮긴다.

이유:
- 슬라이드 2는 *문제만 깊이* 박아야 한다. 답을 미리 흘리면 슬라이드 3가 약해짐.
- ㄴ의 반복 구조 (재검색 → 휘발이 3번 반복되는 그림) 자체가 *"낭비"* 메시지를 시각으로 전달.
- ㄷ의 짧은 캡션 한 줄로 *"우리 일이다"* 신호를 깐다.

## 3. 핵심 인용·카피 후보

### 원문 인용 (권위)
- ✅ 카파시: *"RAG에서는 LLM이 모든 질문에서 지식을 처음부터 다시 발견해야 한다. 어떤 것도 축적되지 않는다."* — 슬라이드 하단 작은 글씨로 박아두기 좋음.
- 카파시: *"NotebookLM, ChatGPT 파일 업로드, 그리고 대부분의 RAG 시스템이 이런 방식으로 작동한다."* — 임원이 아는 도구 이름이 모두 들어감 → 강함.

### excalidraw 작업판에서 발굴한 카피
- ✅ *"단순 요약 → 추출"* — 슬라이드 2에서 *"단순 요약"* 까지만 띄우고, 슬라이드 3에서 *"→ 추출"* 을 받는 구조로 쓰면 슬라이드 간 연결이 강해짐.
- *"지식이 더 이상 문서가 아니라, 엔지니어링 가능한 자산(codebase)으로 바뀐다"* — 슬라이드 2가 아니라 슬라이드 3에 어울림. 여기서는 보류.

### 권장 사용
- **슬라이드 본문 큰 인용:** 카파시 *"NotebookLM, ChatGPT 파일 업로드, 그리고 대부분의 RAG 시스템이 이런 방식으로 작동한다."*
- **슬라이드 2 마지막 한 줄 (다음 슬라이드 연결):** *"단순 요약에서 멈춘다."*

## 4. 슬라이드 2 권장안 (확정안 후보)

```
─────────────────────────────────────────────
                슬라이드 2
─────────────────────────────────────────────

[헤드라인 — 큰 글씨, 한 줄]
같은 질문을 두 번 하면, LLM은 두 번 다 처음부터 시작한다.

[보조 한 줄 — 작은 글씨]
RAG · NotebookLM · ChatGPT 파일 업로드 — 모두 같은 한계.

[중앙 다이어그램 — 옵션 ㄴ]
질문 1 → raw 100문서 재검색 → 답변 → ⨯ 휘발
질문 2 → raw 100문서 재검색 → 답변 → ⨯ 휘발   (똑같은 비용)
질문 3 → raw 100문서 재검색 → 답변 → ⨯ 휘발   (또)

[하단 캡션 — 본부 시나리오 1줄, 회색 작은 글씨]
※ 분기 보고용으로 정리한 분석을, 다음 분기에 다시 처음부터 정리하고 있지 않은가.

[하단 우측 — 원문 인용 작은 글씨]
"NotebookLM, ChatGPT 파일 업로드, 그리고 대부분의 RAG 시스템이 이런 방식으로 작동한다." — Karpathy

[좌하단 작은 글씨, 슬라이드 3 연결]
→ 단순 요약에서 멈춘다.

─────────────────────────────────────────────
```

### 이 슬라이드가 작동하는 방식

1. **헤드라인이 임원의 지난주 경험을 끄집어낸다** — "맞아, 그거 짜증났지."
2. **반복 다이어그램이 '낭비' 감각을 시각으로 박는다** — 같은 비용을 세 번 치르는 그림.
3. **본부 시나리오 한 줄 캡션이 "우리 일이다" 신호를 흘린다** — 슬라이드 4의 깊은 매핑은 아껴둠.
4. **카파시 인용이 권위를 깐다** — 임원이 *"이건 우리만의 문제 아니야"* 라고 인식.
5. **마지막 줄 "단순 요약에서 멈춘다"가 슬라이드 3로 자연스럽게 넘긴다** — 슬라이드 3 첫 줄이 *"→ 추출"* 로 받으면 짝이 맞는다.

## 5. 발표자 멘트 (구두 스크립트)

> *"한 번 생각해보시면, 지난 1년 동안 우리가 LLM에 같은 자료를 몇 번이나 다시 올려서 같은 종류의 질문을 던졌는지 떠오르실 겁니다. **매번 LLM은 새로운 인턴처럼 행동합니다.** 어제 시킨 일을 오늘 다시 설명하고, 같은 자료를 다시 읽힙니다. 그동안 정리해둔 분석은 채팅 로그 어딘가에 사라져 있고, 다음에 같은 답이 다시 나온다는 보장도 없습니다. 이게 RAG 기반 도구의 본질적 한계입니다."*

> 💡 *"새로운 인턴"* 비유는 슬라이드에 박지 말고 구두로만 쓴다. 슬라이드는 카피처럼 보이면 안 된다.

## 6. 임원 Q&A 대비

| 예상 질문 | 답 |
|---|---|
| *"우리 RAG도 캐시 같은 거 있지 않나?"* | "캐시는 같은 질문에 같은 답을 주는 것이고, 여기서 말하는 축적은 *질문 사이의 지식이 누적되는 것* 입니다. 다른 층의 문제입니다." |
| *"파인튜닝하면 되지 않나?"* | "파인튜닝은 무거운 학습 과정이고, 본부 자료처럼 **매주 변하는 데이터**에는 비현실적입니다. LLM Wiki는 학습 없이도 데이터 변화를 즉시 반영합니다." (※ 카파시도 가중치 내재화는 *추가 탐색* 단계로만 언급) |
| *"NotebookLM이 잘 되던데?"* | "NotebookLM은 **단일 세션 안에서**는 잘 됩니다. 세션이 끝나면 그동안 만든 분석은 채팅 안에 남고, 다음 세션은 처음부터 시작합니다 — 바로 그게 슬라이드 2의 그림입니다." |
| *"그럼 그냥 채팅 로그를 잘 정리하면 되는 거 아닌가?"* | "그게 정확히 위키입니다. 단, **사람이 정리하면 유지가 안 되어서 다들 포기합니다** — 그 이유를 슬라이드 5에서 설명드립니다." (슬라이드 5의 "유지비용 0" 메시지로 연결) |
