# 초안 v2 — page 3 작업판 (슬라이드 3)

> **목적:** 슬라이드 3 ("핵심 아이디어 — LLM Wiki란 무엇인가") 를 **하나의 통합 다이어그램**으로 구성한다.
> **사용자 방향성:**
> - Raw / Wiki / Schema 3-Layer 구조
> - *"Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다"* 핵심 비유
> - 운영 사이클 3개 (Ingest / Query / Lint)
> - 보조 파일 (`index.md`, `log.md`)
> 위 4가지를 **하나의 다이어그램에 함께 그려서** 임원이 5초 안에 구조 전체를 파악하게 한다.
> **분량 제약:** 1장. 다이어그램 1개 + 헤드라인 1줄 + 강조 인용 1줄 + 보조 1~2줄.

## 2. 다이어그램 구성 옵션 (이번 페이지의 핵심)

### 옵션 ㄱ — 가로 3-Layer + 사이클 별첨

```
[Raw sources]  →  [Wiki (.md files)]  →  [Schema (CLAUDE.md)]
  (immutable)        (LLM이 작성)         (운영 규칙)

   ───────────────────────────────────────────
   운영:  Ingest  /  Query  /  Lint
   탐색:  index.md  /  log.md
   ───────────────────────────────────────────
```

- **장점:** 단순. 빠르게 그릴 수 있음.
- **단점:** *정적 구조*와 *동적 사이클*이 분리되어 있어, 임원이 "이게 어떻게 돌아가는지" 이해하려면 두 번 봐야 함. **임팩트가 분산됨.**

### 옵션 ㄴ — IDE 비유를 다이어그램 골격으로 (개발 출신 임원에 최적)

```
┌──── "Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다" ────┐
│                                                                  │
│   IDE (Obsidian)        Programmer (LLM)        Codebase (Wiki)  │
│   ┌──────────┐         ┌──────────┐           ┌──────────────┐   │
│   │ 사람이    │ ←view─  │   LLM    │ ──write→  │  .md pages   │   │
│   │ 보는 화면 │         │  Agent   │           │  (요약·엔티티)│   │
│   └──────────┘         └──────────┘           └──────────────┘   │
│                            │  ↑                      │           │
│                       reads│  │navigates             │           │
│                            ↓  │                      ↓           │
│                  ┌──────────────┐         ┌──────────────────┐  │
│                  │ Raw sources  │         │ index.md / log.md│  │
│                  │  (immutable) │         │  (네비게이션)     │  │
│                  └──────────────┘         └──────────────────┘  │
│                            ↑                                     │
│                            │ Schema (CLAUDE.md) — 운영 규칙       │
│                                                                  │
│   세 가지 작업: Ingest (소스 흡수) / Query (질의→답변→축적)         │
│                  Lint (모순·누락 점검)                             │
└──────────────────────────────────────────────────────────────────┘
```

- **장점:** **IDE 비유가 다이어그램 골격 그 자체.** 모든 요소가 비유 위에 매핑됨. 개발 출신 임원에게는 *"이건 친숙한 풍경"* 으로 보임.
- **단점:** 요소가 7개 (IDE/Programmer/Codebase/Raw/Schema/Index·Log/Operations). 시각적 밀도 높음. 폰트와 정렬 정성 필요.

### 옵션 ㄷ — 두 다이어그램 좌·우 분리 (정적 + 동적)

```
[좌: 정적 구조]                      [우: 동적 사이클]
                                      
  Raw   →  Wiki  →  Schema             ┌────────┐
  (불변)   (LLM이      (운영 규칙)        │ Ingest │
            작성)                        └───┬────┘
                                            ↓
              ↑                          ┌────────┐
              │                          │ Query  │
        Obsidian (IDE)                   └───┬────┘
        — 사람이 보는 창 —                  ↓
                                         ┌────────┐
                                         │  Lint  │
                                         └───┬────┘
                                             ↓
                                       ┌─────────────┐
                                       │ index.md    │
                                       │ log.md      │
                                       │ (navigator) │
                                       └─────────────┘

   "Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다."
```

- **장점:** *구조*와 *작동* 분리. 각 다이어그램은 단순.
- **단점:** **사용자 요청 ("하나의 다이어그램")에 어긋남.** 슬라이드도 시각적으로 나뉘어 보임 → 메시지가 둘로 쪼개짐.

### 옵션 ㄹ — IDE 화면 mock + 주석 (가장 구체적·즉시 이해)

```
┌─────────────────────────── Obsidian ──────────────────────────┐
│                                                                 │
│  📁 raw/             📁 wiki/                💬 LLM 채팅         │
│   ├ paper1.pdf        ├ index.md  ←──┐         "이 문서 정리해줘" │
│   ├ meeting.md        ├ log.md       │             │             │
│   └ ...               ├ entity_A.md  │             ↓             │
│                       ├ concept_X.md │  ┌──────────────────┐   │
│  📄 CLAUDE.md         └ ...          │  │   LLM Agent      │   │
│   (Schema)                           └──│  Ingest / Query  │   │
│                                         │  / Lint          │   │
│                                         └──────────────────┘   │
└─────────────────────────────────────────────────────────────────┘

             "Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다."
```

- **장점:** *추상*이 아니라 *실제 작업 화면*처럼 보임. 임원이 "이걸 켜놓고 일하는 거구나" 즉시 이해. excalidraw 작업판의 *"LLM이 만든 지식 구조를 사람이 이해하는 인터페이스"* 카피와 짝이 맞음.
- **단점:** Mock UI 스크린샷 만드는 부담. 이미 본 사람에게는 *"그래서 그게 뭔데"* 가 될 수 있음 (구조 자체는 옵션 ㄴ보다 약함).

### 권장: **옵션 ㄴ (IDE 비유 골격)** 메인.

이유:
- 사용자 요청대로 *"하나의 다이어그램"* 에 4가지 요소를 다 담을 수 있는 구성은 ㄴ뿐.
- IDE/프로그래머/코드베이스 비유가 다이어그램의 *구조 자체*를 정당화함 → 비유와 다이어그램이 분리되지 않음.
- 옵션 ㄹ (Mock UI) 는 매력적이지만 슬라이드 3가 *추상 구조* 슬라이드라는 본래 역할에서 벗어남. **ㄹ은 발표 도중 보충 자료로 쓰는 옵션**으로 남겨둔다.

   IDE (Obsidian)        Programmer (LLM Agent)      Codebase (Wiki)

─────────────────────────────────────────────────────────────────

### 다이어그램이 한 장에 담는 4가지 요소

| 다이어그램 위치 | 담는 내용 |
|---|---|
| 좌·중·우 3개 박스 | **IDE 비유** (Obsidian / LLM / Wiki) — 3-Layer Wiki 부분 |
| 가운데 LLM 박스 안 라벨 | **운영 사이클** (Ingest / Query / Lint) |
| 좌하 (Raw sources) | **3-Layer 중 Raw 레이어** |
| 우하 (index.md/log.md) | **탐색·로그 보조 파일** (RAG 인프라 불필요 신호) |
| 좌하 아래 화살표 (Schema) | **3-Layer 중 Schema 레이어** |
| 다이어그램 전체 골격 | IDE 비유 자체가 시각화됨 |

### 이 한 장이 작동하는 방식

1. **헤드라인이 슬라이드 2를 받는다** — *"단순 요약 → 추출·구조화·누적"*. 임원이 머리 속에서 자동으로 슬라이드 2의 다이어그램과 비교.
2. **다이어그램 골격이 비유 자체** — IDE/프로그래머/코드베이스 3개 박스가 가로로 배치되어, *"이건 개발 환경 같은 거구나"* 가 즉시 인식됨.
3. **운영 사이클이 LLM 박스 안에 숨어 있다** — Ingest/Query/Lint를 별도 영역이 아니라 *프로그래머가 하는 일* 로 표현. 비유와 작동이 분리되지 않음.
4. **index.md/log.md가 우하단에 있다** — RAG 인프라 없이도 작동한다는 *"리스크가 작다"* 신호 (슬라이드 5 비용 0 메시지로 연결).
5. **Schema가 좌하단 아래 화살표로** — 부수적이지만 빠지면 안 되는 요소. *"운영 규칙이 따로 있다 → 이건 즉흥적인 게 아니다"* 신호.
6. **마무리 카피가 임원 톤으로 닫는다** — *"엔지니어링 가능한 자산"*. 슬라이드 4의 본부 매핑으로 자연스럽게 넘어감.

## 4. 발표자 멘트 (구두)

> *"이 그림이 오늘 발표의 본체입니다. 가운데가 LLM 에이전트, 오른쪽이 위키 — 마크다운 파일들의 모음입니다. LLM이 위키를 직접 작성하고 유지합니다. 사람은 왼쪽 Obsidian 화면으로 결과만 봅니다. 즉 **사람은 IDE에서 코드를 보고, LLM이 코드(위키)를 작성합니다.** 카파시 본인의 비유입니다.*
>
> *LLM이 하는 일은 세 가지입니다. **Ingest** — 새 자료가 들어오면 흡수해서 관련 페이지를 한꺼번에 업데이트. **Query** — 질문에 답하고, 그 답을 다시 위키로 적립. **Lint** — 주기적으로 모순·누락을 점검.*
>
> *오른쪽 아래 두 파일이 핵심입니다. `index.md`는 위키 전체 카탈로그, `log.md`는 작업 로그. **이 두 파일만으로 100개 문서·40만 단어 규모까지 별도 RAG 인프라 없이 작동합니다.** 즉 시작 비용이 거의 0입니다."*

> 💡 *"별도 RAG 인프라 없이"* 한 줄은 슬라이드 5 (4~6주 파일럿, 비용 0) 의 핵심 근거가 된다. 슬라이드 3에서 한 번 깔고 슬라이드 5에서 회수하는 구조.

## 5. 임원 Q&A 대비

| 예상 질문 | 답 |
|---|---|
| *"위키를 LLM이 다 쓴다고? 사람이 검토 안 하나?"* | "검토는 가능합니다. 카파시는 *'필요하면 사람이 검토 단계에 참여'* 로 명시했습니다. 본부 적용 시 첫 1~2주는 모든 ingest를 사람이 검토하는 모드로 시작하면 됩니다." |
| *"Obsidian 꼭 써야 하나? 우리는 Confluence/Notion인데."* | "Obsidian은 *마크다운 파일을 시각화해주는 IDE 역할*입니다. 마크다운 파일 자체가 본체이고, Obsidian은 보조 도구입니다. Confluence·Notion에 마크다운 파일을 import하는 방식도 가능합니다." |
| *"`CLAUDE.md`가 뭔데? 따로 만들어야 하나?"* | "LLM에게 *'이 위키는 이렇게 운영하라'* 고 알려주는 한 페이지짜리 운영 매뉴얼입니다. 본부 첫 적용 시 1주일 정도면 초안 완성, 운영하면서 다듬어 나가는 형태입니다." |
| *"`index.md`로만 정말 100문서까지 충분한가?"* | "카파시 본인이 *~100개 문서, ~40만 단어 규모에서 별도 RAG 없이 작동* 한다고 명시했습니다. 본부 파일럿 규모 (4~6주, 작은 영역 1개) 라면 이 범위 안에 들어옵니다." |
| *"이게 그냥 NotebookLM 좀 잘 쓰는 거 아닌가?"* | "결정적 차이는 **Wiki 레이어가 따로 있다**는 점입니다. NotebookLM은 raw 문서만 가지고 매번 답을 만들고, 답은 채팅 안에 사라집니다. 여기는 raw와 답 사이에 *유지되는 마크다운 파일들*이 있고, 좋은 답은 다시 그 파일에 적립됩니다 — 슬라이드 2의 그림과 다른 점입니다." |

## 부록 — 다이어그램 그릴 때 정성 포인트

> 옵션 ㄴ은 정렬·여백·폰트가 결과를 좌우한다. 손으로 빠르게 그리지 말고 정렬을 잡고 그릴 것.

1. **3개 메인 박스 (IDE / Programmer / Wiki) 는 같은 크기, 같은 높이.** 시각적 평행이 비유의 평행을 강화한다.
2. **화살표 라벨 (`view` / `write` / `reads` / `navigates`) 은 동사로 통일.** 명사형 (`viewing`) 금지.
3. **가운데 LLM 박스만 약간 강조 (배경색·굵은 테두리).** 이 슬라이드의 주인공은 LLM이다.
4. **`Schema (CLAUDE.md)` 는 좌하단에 작게.** 중요하지만 메인 흐름은 아니다.
5. **카파시 인용은 다이어그램과 충분히 떨어진 위치에 배치.** 인용이 그림을 가리지 않게.
6. **하단 마무리 카피 ("엔지니어링 가능한 자산") 는 회색·작게.** 헤드라인과 경쟁하지 않게.
