> **목적:** Software 3.0 프레임 (지시문이 프로그램이다) 을 슬라이드 3에 통합.

## 핵심 개념 정리 (발표자 구두용)

> 소프트웨어에는 세 가지 방식이 있다.
>
> **첫 번째:** 사람이 코드를 작성한다. 규칙이 명시적이다.
> **두 번째:** 사람이 데이터를 모아 모델을 학습시킨다. 가중치가 프로그램이 된다. 규칙이 암묵적으로 학습된다.
> **세 번째:** 이미 학습된 LLM에 자연어로 지시를 내린다. **지시문(prompt)이 새로운 프로그램이다.**
>
> LLM Wiki의 "운영 규칙 파일"이 바로 그 지시문이다.
> LLM은 그 지시문대로 위키를 빌드하고, 유지하고, 점검한다.

## 화면 구성 옵션

### 옵션 ① — 3단 타임라인 + 다이어그램 [권장]

```
─────────────────────────────────────────────────────────────────
                        슬라이드 3
─────────────────────────────────────────────────────────────────

[헤드라인]
소프트웨어의 세 번째 방식 — 지시문이 프로그램이다.

[3단 미니 타임라인 — 작은 글씨, 가로 배치]

  ① 코드가 프로그램         ② 가중치가 프로그램         ③ 지시문이 프로그램
    (전통 소프트웨어)           (머신러닝)                (LLM 시대)  ← 지금 여기
    사람이 규칙을 작성          데이터가 규칙을 학습         자연어가 규칙을 지시

[중앙 다이어그램]

   Obsidian              LLM 에이전트              인하우스 위키
  (보기 화면)
   ┌──────────┐          ┌────────────────┐       ┌──────────────┐
   │ 사람이    │ ←view─── │  Ingest        │ ─write▶  주제별 .md  │
   │ 보는 화면 │          │  Query         │       │  요약·관계   │
   │          │          │  Lint          │       │  개념·비교   │
   └──────────┘          └────────────────┘       └──────────────┘
                           │          ↑                  │
                      reads│          │navigates         ↓
                           ▼          │         ┌──────────────────┐
                    ┌──────────────┐  │         │  색인 · 작업로그  │
                    │  원본 자료    │──┘         │  (탐색 + 이력)   │
                    │  (원본 보존) │             └──────────────────┘
                    └──────────────┘
                           ↑
                    운영 규칙 파일  ← 이게 지시문 = 프로그램
                    (LLM에게 줄 정리 방식·기준)

[핵심 인용]
"Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다." — Karpathy

─────────────────────────────────────────────────────────────────
```

> **이 구성의 강점:**
> - 타임라인 3줄이 "왜 지금인가" 를 5초 안에 전달
> - 다이어그램의 "운영 규칙 파일" 이 지시문 = 프로그램임을 타임라인이 미리 설명
> - Karpathy 인용이 타임라인 + 다이어그램을 하나의 비유로 묶음

## 권장: 옵션 ①

이유:
- 슬라이드 3이 5장 중 가장 중요하다면, 화면 자체가 논거를 담아야 한다
- 타임라인 3줄은 작게 배치하면 슬라이드 밀도에 큰 영향 없음
- "운영 규칙 파일 = 지시문" 연결이 다이어그램을 훨씬 설득력 있게 만듦
- 슬라이드 5 "왜 지금인가" 논거와 연결됨 (세 번째 패러다임이 지금 성숙)

## 발표자 멘트 (옵션 ① 기준)

> *"소프트웨어가 작동하는 방식이 역사적으로 세 번 바뀌었습니다. 첫 번째는 우리가 아는 방식 — 사람이 코드를 작성하고 컴퓨터가 실행합니다. 두 번째는 머신러닝 — 사람이 데이터를 모으면 모델이 규칙을 스스로 학습합니다. 가중치가 프로그램이 됩니다.*
>
> *세 번째가 지금 일어나고 있습니다. 코드도 아니고, 학습도 아닙니다. 이미 학습된 LLM에 자연어로 지시를 내립니다. **그 지시문이 프로그램입니다.** 카파시가 말한 'Wiki는 코드베이스'라는 비유가 여기서 나옵니다.*
>
> *이 다이어그램을 보시면, 가운데 LLM 에이전트가 세 가지 일을 합니다. 새 자료를 흡수하고 (Ingest), 질문에 답하면서 위키를 쌓고 (Query), 주기적으로 점검합니다 (Lint). 그 아래 보이는 운영 규칙 파일 — 이게 LLM에게 내리는 지시문입니다. 이 지시문 하나가 지식 빌드 시스템 전체를 구동합니다."*

## 임원 Q&A 대비

| 예상 질문 | 답 |
|---|---|
| *"세 번째 방식이 다른 AI 도구랑 뭐가 달라?"* | "ChatGPT·Claude도 세 번째 방식입니다. LLM Wiki는 그 위에 한 층을 더 올린 것 — 지시문으로 지식 빌드 시스템을 운영합니다. 도구가 아니라 시스템입니다." |
| *"지시문(프롬프트)이 프로그램이면 누가 쓰나?"* | "처음 한 번 씁니다. 이후 LLM이 스스로 위키를 유지합니다. 카파시 gist 자체가 그 지시문의 예시입니다." |
| *"위키를 LLM이 다 쓴다고? 사람이 검토 안 하나?"* | "검토 단계를 넣을 수 있습니다. 첫 1~2주는 모든 결과를 사람이 확인하는 모드로 시작하면 됩니다." |
| *"이게 NotebookLM이랑 뭐가 다른가?"* | "NotebookLM은 raw 문서만 보고 매번 처음부터 답합니다. 여기는 raw와 답 사이에 유지되는 위키가 있고, 좋은 답이 다시 그 위키에 쌓입니다." |
