## 슬라이드 2 — 문제 정의

### 화면 구성 (위 → 아래)

**① 헤드라인 (큰 글씨)**

> 자료는 쌓이고 있다. 이해는 쌓이고 있는가?

**② 중앙 다이어그램 — 개인 업무·생각 정리 시나리오**

```
① 반복
   오늘   "AI 투자 트렌드 정리해줘" → ChatGPT에 배경부터 다시 설명 → 정리 → ⨯ 휘발
   다음 달 같은 주제 다시 궁금     → 배경부터 다시 설명              → 정리 → ⨯ 또 휘발
                                                         ← 지난달에 정리한 게 어딘가 있는데

② 진화 없음
   1월  어떤 주제에 대해 A라고 생각 → 메모앱에 저장
   4월  공부 더 하고 나서 B로 바뀜  → 또 다른 메모에 저장
   7월  다시 꺼내보니               → ⨯ A인지 B인지, 왜 바뀌었는지 모름
                                         생각의 흐름이 어디에도 없음

③ 관계 무지
   책에서 읽은 아이디어     ─┐
   지난주 미팅에서 나온 얘기 ─┤  →  ⨯ 연결되는 줄 모름
   어제 읽은 뉴스 기사      ─┘       나중에 "아 그게 그거였네" 하고 지나침
```

**③ 하단 우측 — Karpathy 인용 (작은 글씨)**

> "이전에 했던 작업이 남지 않기 때문에, 매번 동일한 과정을 반복하게 된다." — Karpathy

### 발표자 멘트

> *"개인 수준에서 먼저 보겠습니다. 관심 있는 주제를 ChatGPT에 물어봅니다. 다음 달에 같은 주제가 다시 궁금해집니다. 또 처음부터 설명합니다. 지난달에 정리한 게 어딘가에 있는데, 찾기보다 다시 묻는 게 빠릅니다.*
>
> *생각도 마찬가지입니다. 1월에 A라고 생각해서 메모했고, 4월에 공부를 더 하고 나서 B로 바뀌었는데, 7월에 다시 꺼내보면 A인지 B인지, 왜 바뀌었는지 흔적이 없습니다.*
>
> *그리고 연결. 책에서 읽은 아이디어, 지난주 미팅에서 나온 얘기, 어제 읽은 기사 — 사실 세 개가 연결되는데 그 연결을 놓칩니다. 나중에 '아, 그게 그거였네' 하고 지나칩니다.*
>
> *이건 도구의 문제가 아닙니다. ChatGPT를 더 잘 쓴다고 해결되지 않습니다. **내 이해가 쌓이는 곳이 없다는 구조의 문제입니다.***"*

## 슬라이드 3 — 핵심 아이디어

### 화면 구성 (위 → 아래)

**① 헤드라인 (큰 글씨)**

> 단순 요약 → 추출 · 구조화 · 누적.

**② 중앙 다이어그램 — 5박스 구조 + 운영 사이클**

```
   Obsidian              LLM 에이전트              인하우스 위키
  (보기 화면)
   ┌──────────┐          ┌────────────────┐       ┌──────────────┐
   │ 사람이    │ ←view─── │  Ingest        │ ─write▶  주제별 .md  │
   │ 보는 화면 │          │  Query         │       │  요약·관계   │
   │          │          │  Lint          │       │  개념·비교   │
   └──────────┘          └────────────────┘       └──────────────┘
                           │          ↑                  │
                      reads│          │navigates         │
                           ▼          │                  ▼
                    ┌──────────────┐      ┌──────────────────┐
                    │  원본 자료    │      │  색인 · 작업로그  │
                    │  (원본 보존) │      │  (탐색 + 이력)   │
                    └──────────────┘      └──────────────────┘
                           ↑
                      운영 규칙 파일 (CLAUDE.md)
                      — LLM에게 줄 정리 방식·기준
```

- **Ingest** — 새 자료 → 관련 위키 페이지 자동 업데이트
- **Query** — 질문에 답하고, 좋은 답은 다시 위키로 적립
- **Lint** — 주기적으로 모순·누락·오래된 내용 점검

**③ 핵심 인용 (다이어그램 하단, 강조)**

> "Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다." — Karpathy

**④ 마무리 한 줄 (하단 회색)**

> 지식이 문서가 아니라, 엔지니어링 가능한 자산으로 바뀐다.

---

### 발표자 멘트

> *"이 그림이 오늘 발표의 본체입니다. 오른쪽이 인하우스 위키 — 주제별로 정리된 마크다운 파일들의 모음입니다. LLM이 이 위키를 직접 작성하고 유지합니다. 사람은 결과만 봅니다.*
>
> *LLM이 하는 일은 세 가지입니다. 새 자료가 들어오면 흡수해서 관련 페이지를 업데이트하고 (Ingest), 질문에 답하면서 그 답을 다시 위키로 쌓고 (Query), 주기적으로 모순이나 누락을 점검합니다 (Lint).*
>
> *카파시는 이걸 이렇게 표현했습니다. '위키는 코드베이스고, LLM은 그 코드베이스를 작성하는 프로그래머다.' 중요한 점은 — 이게 별도 서버나 RAG 인프라가 필요한 구조가 아닙니다. 마크다운 파일 폴더 하나로 시작합니다. 그래서 파일럿 비용이 거의 0입니다."*

## 슬라이드 4 — 그래서, 우리가 뭘 할 수 있게 됐는데?

### 화면 구성 (위 → 아래)

**① 헤드라인 (큰 글씨)**

> 그래서, 우리가 뭘 할 수 있게 됐는데?

**② 부제 (작은 글씨, 한 줄)**

> 지금까지 못 했던 4가지가 가능해진다.

**③ 중앙 — capability 4박스 (2×2 그리드)**

```
┌─────────────────────────────────┬─────────────────────────────────┐
│ ①  자료가 자료를 정리한다         │ ②  질문이 자산이 된다             │
│    (Ingest)                     │    (Query — closed loop)        │
│  ─────────────                  │  ─────────────                  │
│  매일 들어오는 셀사이드 PDF 50건  │  "지난 분기 반도체 섹터           │
│  → 종목 / 섹터 / 매크로 페이지로  │   의견 변화 정리해줘"             │
│     자동 분배 · 갱신             │   → 답변                         │
│                                  │   → 답변이 다시 위키로 적립        │
│  사람이 분류·요약할 필요 없음.    │                                  │
│                                  │  다음 질문은 더 빠르고 더 좋아진다.│
├─────────────────────────────────┼─────────────────────────────────┤
│ ③  위키가 스스로 깨끗해진다       │ ④  사람의 일이 바뀐다             │
│    (Lint)                       │                                  │
│  ─────────────                  │  ─────────────                  │
│  한 달 전 강세 의견              │  애널리스트:                      │
│  vs 지금 약세 의견               │   수집·요약·분류 → 분석·판단·소통  │
│  → 모순 / 변화 자동 표시         │                                  │
│                                  │  운용역:                         │
│  오래된 코멘트 플래그.            │   "누가 봤더라?" 검색 → 질문하기  │
│  누락된 종목·섹터 갭 점검.        │                                  │
└─────────────────────────────────┴─────────────────────────────────┘
```

**④ 하단 강조 인용 (가운데, 굵게)**

> **"운용역이 질문하는 행위 자체가 본부 인하우스 자료를 키운다."**

**⑤ 하단 회색 1줄 (오른쪽 끝)**

> ※ 첫 파일럿은 컴플라이언스 부담이 가장 낮은 외부 발행 리서치(셀사이드 / 거시)에서 시작 — 다음 슬라이드.

### 발표자 멘트

> *"앞 슬라이드에서 구조를 보셨습니다. 그러면 이게 본부 일에서 구체적으로 뭘 가능하게 하는가 — 네 가지입니다.*
>
> *첫째, **자료가 자료를 정리합니다.** 매일 GS·모건스탠리·JPM·국내 증권사에서 셀사이드 PDF가 쏟아집니다. 지금은 받은 사람이 읽고 끝납니다. LLM이 흡수하면 종목·섹터·매크로 페이지로 자동 분배·갱신됩니다. 사람이 분류할 필요가 없어집니다.*
>
> *둘째, 이게 가장 중요합니다 — **질문이 자산이 됩니다.** 운용역이 '지난 분기 반도체 섹터 의견 변화 정리해줘' 라고 물으면 답이 옵니다. 그런데 그 답이 채팅 로그로 사라지지 않고, **다시 위키에 적립됩니다.** 다음 사람이 같은 질문을 하면 더 빠르고 더 좋은 답이 나옵니다. **운용역이 질문하는 행위 자체가 본부 자료를 키우는 구조입니다.***
>
> *셋째, **위키가 스스로 깨끗해집니다.** 한 달 전 강세 의견과 지금 약세 의견이 충돌하면 표시됩니다. 오래된 코멘트, 빠진 섹터 — 자동 점검됩니다. 위키가 시간 지나면서 망가지던 이유가 유지보수였는데, 그걸 LLM이 가져갑니다.*
>
> *그리고 넷째, **사람의 일이 바뀝니다.** 애널리스트는 수집·요약·분류라는 반복 작업에서 풀려 분석·판단·소통에 집중합니다. 운용역은 '그 리포트 누가 봤더라?' 검색에서 풀려 질문을 던지는 사람이 됩니다. 즉 위키가 애널리스트를 대체하는 게 아니라, **애널리스트가 더 위로 올라갑니다.***"*

### 임원 Q&A 대비

| 예상 질문 | 답 |
|---|---|
| *"4개 다 한 번에 되는 건가?"* | "①②는 1일 차부터. ③ Lint는 자료가 어느 정도 쌓인 후(2~3주 차) 의미가 생깁니다. ④는 결과물입니다 — 4주차 보고에서 변화가 보일 영역." |
| *"답이 틀리면? closed loop가 오히려 오류를 적립하는 거 아닌가?"* | "모든 답은 raw 소스로 인용·역추적됩니다. 적립 단계에서 검토 모드를 둘 수 있고, Lint가 모순을 드러냅니다 — 오류가 *감춰지는* 게 아니라 *드러나는* 구조." |
| *"Bloomberg / FactSet 대체?"* | "**대체 아닙니다. 보조입니다.** Bloomberg·FactSet에서 받는 PDF·메일·메모를 마크다운 한 겹으로 정리할 뿐, 두 시스템 다 그대로 사용." |
| *"애널리스트들이 자기 일 뺏긴다고 느끼지 않을까?"* | "*반복 정리*만 LLM이 가져갑니다. 분석·판단·소통은 사람의 영역. 게다가 위키가 애널리스트 작업물의 색인이 되어, 본인 결과물이 더 자주 인용됩니다." |
| *"퀀트팀 모델 노트는?"* | "1주차 검증을 거기서 합니다 — 흩어진 ipynb·슬랙 메모가 본부 자료로 합류. 검증 후 외부 리서치 영역으로 확대." |

### 슬라이드 5와의 연결

박스 ①②③ 이 *시스템 효용*, ④ 가 *조직 효용*. 슬라이드 5에서 이 4개 capability 가 *"PM이 하는 4가지 일"* 로 자연스럽게 받아진다 — 효용을 행동으로 거울 비추는 형태.

## 슬라이드 5 — 이제, 우리가 PM입니다.

### 화면 구성 (위 → 아래)

**① 헤드라인 (큰 글씨)**

> 이제, 우리가 PM입니다.

**② 부제 — 슬라이드 3 비유 회수·확장 (작은 글씨, 한 줄)**

> LLM = 프로그래머 · .md = 코드 · Wiki = 코드베이스 · **우리 = PM**

**③ 중앙 좌측 — PM 사이클 다이어그램**

```
   ┌─────────────────┐
   │   나 (PM)         │
   │   · 주제 정의     │
   │   · 자료 투입     │
   │   · 질문 / 검토   │
   └────────┬────────┘
            │ 디렉션 · 자료
            ▼
   ┌─────────────────┐
   │  LLM (프로그래머) │
   │   · 읽고          │
   │   · 정리하고      │
   │   · 점검한다      │
   └────────┬────────┘
            │ .md 산출물
            ▼
   ┌─────────────────┐
   │  Wiki (코드베이스)│
   │   계속 자란다     │
   └─────────────────┘
```

**④ 중앙 우측 — PM이 하는 4가지 (= 오늘 시작하는 법)**

```
①  무엇을 담을지 정한다
    "내 커버 종목" / "매크로 노트" / "모델 기록" / "분기 보고 자료" …

②  자료를 던져준다
    매일 받는 PDF · 메모 · 메일을 폴더에 떨군다.

③  질문하고 검토한다
    "지난 분기 의견 흐름 정리" / "모순 점검" / "갱신된 페이지 요약"
    좋은 답은 위키로 적립, 잘못된 건 수정 지시.

④  한 번만 방향을 잡아둔다
    CLAUDE.md 한 장 — 정리 방식 · 톤 · 강조 기준.
    한 번 박으면 LLM이 따른다.
```

**⑤ 하단 강조 (큰 글씨 한 줄, 가운데)**

> **코딩하지 않는다. 폴더 하나에서, 오늘부터.**

**⑥ 마무리 인용 (우측 하단, 작게)**

> *"새로운 종류의 제품이 가능해졌다."* — Karpathy

### 발표자 멘트

> *"마지막입니다. 지금까지 보신 그림에 한 조각만 더 얹겠습니다.*
>
> *카파시는 이렇게 말했습니다 — Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다. 그러면 한 자리가 비어 있습니다. **그 코드베이스를 가진 프로젝트의 PM은 누구인가.** 그게 우리입니다.*
>
> *PM이 하는 일은 네 가지입니다. 첫째, 무엇을 담을지 정합니다 — 내가 커버하는 종목, 매크로 노트, 모델 기록, 무엇이든. 둘째, 자료를 던져줍니다 — 매일 받는 PDF·메모·메일을 폴더에 떨굽니다. 셋째, 질문하고 검토합니다 — 지난 분기 의견 흐름 정리해달라, 모순 점검해달라. 넷째, 한 번만 방향을 잡아둡니다 — CLAUDE.md 한 장. *어떤 톤으로, 무엇을 강조해서 정리할지* 한 번 박아두면 LLM이 따릅니다.*
>
> *중요한 건 — **코딩하지 않는다**는 점입니다. 카파시가 바이브코딩이라는 말을 만든 사람인데, 본질은 *"코드 못 짜도 도메인 알면 만들 수 있다"* 입니다. 위키도 똑같습니다. **본부에 계신 분들은 이미 도메인 전문가입니다. 그게 PM 자격입니다.***
>
> *시작에 필요한 건 폴더 하나입니다. 본부 차원 합의도, 예산도, 결정도 필요 없습니다. 각자 자기 영역에서, 오늘부터 시작하실 수 있습니다. **카파시 본인이 말한 대로 — 새로운 종류의 제품이 가능해진 시점입니다.** 본부 안에서도 그 시점이 시작되었으면 좋겠다는 게 오늘 발표의 결론입니다."*
