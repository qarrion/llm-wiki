## 슬라이드 2 — 문제 정의

### 화면 구성 (위 → 아래)

**① 헤드라인 (큰 글씨)**

> 자료는 쌓이고 있다. 이해는 쌓이고 있는가?

**② 중앙 다이어그램 — 개인 업무·생각 정리 시나리오**

```
① 반복
   오늘   "AI 투자 트렌드 정리해줘" → ChatGPT에 배경부터 다시 설명 → 정리 → ⨯ 휘발
   다음 달 같은 주제 다시 궁금     → 배경부터 다시 설명              → 정리 → ⨯ 또 휘발
                                                         ← 지난달에 정리한 게 어딘가 있는데

② 진화 없음
   1월  어떤 주제에 대해 A라고 생각 → 메모앱에 저장
   4월  공부 더 하고 나서 B로 바뀜  → 또 다른 메모에 저장
   7월  다시 꺼내보니               → ⨯ A인지 B인지, 왜 바뀌었는지 모름
                                         생각의 흐름이 어디에도 없음

③ 관계 무지
   책에서 읽은 아이디어     ─┐
   지난주 미팅에서 나온 얘기 ─┤  →  ⨯ 연결되는 줄 모름
   어제 읽은 뉴스 기사      ─┘       나중에 "아 그게 그거였네" 하고 지나침
```

**③ 하단 우측 — Karpathy 인용 (작은 글씨)**

> "이전에 했던 작업이 남지 않기 때문에, 매번 동일한 과정을 반복하게 된다." — Karpathy

**④ 슬라이드 3 연결 한 줄 (좌하단)**

> → 단순 요약에서 멈춘다.

---

### 발표자 멘트

> *"개인 수준에서 먼저 보겠습니다. 관심 있는 주제를 ChatGPT에 물어봅니다. 다음 달에 같은 주제가 다시 궁금해집니다. 또 처음부터 설명합니다. 지난달에 정리한 게 어딘가에 있는데, 찾기보다 다시 묻는 게 빠릅니다.*
>
> *생각도 마찬가지입니다. 1월에 A라고 생각해서 메모했고, 4월에 공부를 더 하고 나서 B로 바뀌었는데, 7월에 다시 꺼내보면 A인지 B인지, 왜 바뀌었는지 흔적이 없습니다.*
>
> *그리고 연결. 책에서 읽은 아이디어, 지난주 미팅에서 나온 얘기, 어제 읽은 기사 — 사실 세 개가 연결되는데 그 연결을 놓칩니다. 나중에 '아, 그게 그거였네' 하고 지나칩니다.*
>
> *이건 도구의 문제가 아닙니다. ChatGPT를 더 잘 쓴다고 해결되지 않습니다. **내 이해가 쌓이는 곳이 없다는 구조의 문제입니다.***"*

## 슬라이드 3 — 핵심 아이디어

### 화면 구성 (위 → 아래)

**① 헤드라인 (큰 글씨)**

> 단순 요약 → 추출 · 구조화 · 누적.

**② 중앙 다이어그램 — 5박스 구조 + 운영 사이클**

```
   Obsidian              LLM 에이전트              인하우스 위키
  (보기 화면)
   ┌──────────┐          ┌────────────────┐       ┌──────────────┐
   │ 사람이    │ ←view─── │  Ingest        │ ─write▶  주제별 .md  │
   │ 보는 화면 │          │  Query         │       │  요약·관계   │
   │          │          │  Lint          │       │  개념·비교   │
   └──────────┘          └────────────────┘       └──────────────┘
                           │          ↑                  │
                      reads│          │navigates         │
                           ▼          │                  ▼
                    ┌──────────────┐      ┌──────────────────┐
                    │  원본 자료    │      │  색인 · 작업로그  │
                    │  (원본 보존) │      │  (탐색 + 이력)   │
                    └──────────────┘      └──────────────────┘
                           ↑
                      운영 규칙 파일 (CLAUDE.md)
                      — LLM에게 줄 정리 방식·기준
```

- **Ingest** — 새 자료 → 관련 위키 페이지 자동 업데이트
- **Query** — 질문에 답하고, 좋은 답은 다시 위키로 적립
- **Lint** — 주기적으로 모순·누락·오래된 내용 점검

**③ 핵심 인용 (다이어그램 하단, 강조)**

> "Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다." — Karpathy

**④ 마무리 한 줄 (하단 회색)**

> 지식이 문서가 아니라, 엔지니어링 가능한 자산으로 바뀐다.

---

### 발표자 멘트

> *"이 그림이 오늘 발표의 본체입니다. 오른쪽이 인하우스 위키 — 주제별로 정리된 마크다운 파일들의 모음입니다. LLM이 이 위키를 직접 작성하고 유지합니다. 사람은 결과만 봅니다.*
>
> *LLM이 하는 일은 세 가지입니다. 새 자료가 들어오면 흡수해서 관련 페이지를 업데이트하고 (Ingest), 질문에 답하면서 그 답을 다시 위키로 쌓고 (Query), 주기적으로 모순이나 누락을 점검합니다 (Lint).*
>
> *카파시는 이걸 이렇게 표현했습니다. '위키는 코드베이스고, LLM은 그 코드베이스를 작성하는 프로그래머다.' 중요한 점은 — 이게 별도 서버나 RAG 인프라가 필요한 구조가 아닙니다. 마크다운 파일 폴더 하나로 시작합니다. 그래서 파일럿 비용이 거의 0입니다."*

### 임원 Q&A 대비

| 예상 질문 | 답 |
|---|---|
| *"위키를 LLM이 다 쓴다고? 사람이 검토 안 하나?"* | "검토 단계를 넣을 수 있습니다. 첫 1~2주는 모든 결과를 사람이 확인하는 모드로 시작하면 됩니다." |
| *"Obsidian을 꼭 써야 하나? 우리는 Notion인데."* | "Obsidian은 보조 도구입니다. 마크다운 파일 자체가 본체이고, Notion에 import하는 방식도 가능합니다." |
| *"이게 NotebookLM이랑 뭐가 다른가?"* | "NotebookLM은 raw 문서만 보고 매번 처음부터 답합니다. 여기는 raw와 답 사이에 유지되는 위키가 있고, 좋은 답이 다시 그 위키에 쌓입니다." |
| *"100개 문서 이상이면?"* | "카파시 본인이 ~100개 문서·40만 단어까지 별도 인프라 없이 작동한다고 명시했습니다. 파일럿 규모는 이 안에 들어옵니다." |
