## 슬라이드 6 — 이제, 우리가 PM입니다

### 화면 구성 (위 → 아래)

**① 헤드라인 (큰 글씨)**

> 이제, 우리가 PM입니다.

**② 부제 — 슬라이드 4 비유 회수·확장 (한 줄)**

> LLM = 프로그래머 · .md = 코드 · Wiki = 코드베이스 · **우리 = PM**

**③ 중앙 본체 — PM이 하는 4가지 (큰 글씨·중앙·이게 주인공)**

```
①  무엇을 담을지 정한다
    "내 커버 종목" / "매크로 노트" / "모델 기록" / "분기 보고 자료" …

②  자료를 던져준다
    매일 받는 PDF · 메모 · 메일을 폴더에 떨군다.

③  질문하고 검토한다
    "지난 분기 의견 흐름 정리" / "모순 점검" / "갱신된 페이지 요약"
    좋은 답은 위키로 적립, 잘못된 건 수정 지시.

④  한 번만 방향을 잡아둔다
    CLAUDE.md 한 장 — 정리 방식 · 톤 · 강조 기준.
    한 번 박으면 LLM이 따른다.
```

**④ 우측 또는 하단 — PM 사이클 (회색·작게·보조)**

```
   나(PM) ─ 디렉션 · 자료 ─▶ LLM(프로그래머) ─ .md 산출 ─▶ Wiki(자란다)
     ▲                                                          │
     └──────────────────────  검토 · 다음 질문  ──────────────────┘
```

**⑤ 강조 (큰 글씨 한 줄, 가운데)**

> **코딩하지 않는다. 폴더 하나에서, 오늘부터.**

**⑥ 본부장 자리 (작은 박스, 회색, 부탁 톤 — 결정 박스 아님)**

```
※ 본부장님께:
  분기 한 번, 본부 안에서 위키가 어떻게 자라고 있는지
  보여드리는 자리만 부탁드립니다.
  결정해주실 일은 없습니다 — 보여드리는 자리입니다.
```

**⑦ 마무리 인용 (우측 하단, 작게)**

> *"새로운 종류의 제품이 가능해졌다."* — Karpathy

### 발표자 멘트

> *"마지막입니다. 지금까지 보신 그림에 한 조각만 더 얹겠습니다.*
>
> *카파시는 이렇게 말했습니다 — Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다. 그러면 **한 자리가 비어 있습니다. 그 코드베이스를 가진 프로젝트의 PM은 누구인가.** 그게 우리입니다.*
>
> *PM이 하는 일은 네 가지입니다. 첫째, **무엇을 담을지 정합니다** — 내가 커버하는 종목, 매크로 노트, 모델 기록, 무엇이든. 둘째, **자료를 던져줍니다** — 매일 받는 PDF·메모·메일을 폴더에 떨굽니다. 셋째, **질문하고 검토합니다** — 지난 분기 의견 흐름 정리해달라, 모순 점검해달라. 넷째, **한 번만 방향을 잡아둡니다** — CLAUDE.md 한 장. *어떤 톤으로, 무엇을 강조해서 정리할지* 한 번 박아두면 LLM이 따릅니다.*
>
> *중요한 건 — **코딩하지 않는다** 는 점입니다. 카파시가 *바이브코딩* 이라는 말을 만든 사람인데, 본질은 *'코드 못 짜도 도메인 알면 만들 수 있다'* 입니다. 위키도 똑같습니다. **본부에 계신 분들은 이미 도메인 전문가입니다. 그게 PM 자격입니다.***
>
> *시작에 필요한 건 폴더 하나입니다. 본부 차원 합의도, 예산도, 결정도 필요 없습니다. 각자 자기 영역에서, 오늘부터 시작하실 수 있습니다.*
>
> ***본부장님께는** 분기 한 번 정도, 본부 위키가 어떻게 자라고 있는지 보여드리는 자리만 마련하면 충분할 것 같습니다. 결정해주실 일은 없습니다 — 보여드리는 자리입니다. 시작해보고, 자라는 모습이 의미 있어 보이면 그때 같이 더 크게 갈지 보면 됩니다.*
>
> ***카파시 본인이 말한 대로 — 새로운 종류의 제품이 가능해진 시점입니다.** 본부 안에서도 그 시점이 시작되었으면 좋겠다는 게 오늘 발표의 결론입니다."*

### 임원 Q&A 대비

| 예상 질문 | 답 |
|---|---|
| *"PM이라는 표현이 거창하지 않나?"* | "거창한 게 아니라 익숙한 역할입니다. 코딩 안 하고 디렉션 주는 사람 — 본부 임원·운용역 일상에 가까운 역할입니다." |
| *"PM 역할이 추가 부담 아닌가?"* | "핵심 일은 ① 주제 정의 + ④ 한 번 방향 잡기. ②③은 일상 업무에 자연스럽게 녹습니다. 분기마다 한 번 리뷰만 추가됩니다." |
| *"사람마다 따로 위키 만들면 본부가 분산되지 않나?"* | "각자 자기 위키부터 시작이 정확합니다. *공유* 는 나중 단계 — 합쳐도 되고 안 합쳐도 됩니다. *분산이 진입 장벽을 낮춥니다.*" |
| *"어디서 시작하나? 어떤 도구?"* | "Cursor / Claude Code / Obsidian 어느 쪽이든. 핵심은 폴더와 .md, 도구는 부차적. 막히면 퀀트팀에 문의해주시면 됩니다." |
| *"검토 부담은?"* | "검토가 PM 일의 일부. 처음 1~2주 모든 산출물 확인, 신뢰 쌓이면 샘플링 — 슬라이드 4의 *Lint* 동작이 그 보조입니다." |
| *"퀀트팀은 뭘 하나?"* | "**한 가지 약속을 드립니다.** 외부 발행 리서치(셀사이드/거시) 한 영역은 우리가 시범으로 깔아 본부에 공유합니다. 본부 내 첫 사례 + 막히는 분 옆에서 도움." |
| *"이게 결국 안 쓰면?"* | "사라지는 건 폴더 하나. 본부 시스템·인프라 안 건드립니다. **롤백 자유.**" |
| *"본부장은 뭘 하시나?"* | "결정해주실 일은 없습니다. 분기 한 번 진척 보여드리는 자리만 부탁드립니다. 자라는 모습이 의미 있어 보이면 그때 같이 더 크게 갈지 보면 됩니다." |
