## 슬라이드 3 — 문제 정의 : 자료는 쌓이고 있다. 이해는 쌓이고 있는가?

### 화면 구성 (위 → 아래)

**① 헤드라인 (큰 글씨)**

> **문제 정의 :** 자료는 쌓이고 있다. 이해는 쌓이고 있는가?

**② 한 줄 통증 (헤드라인 직후, 강조)**

> 매번 업로드한 자료에서 지식을 *다시 발견* 할 뿐, **축적되지 않는다.**
> 현재의 **RAG · NotebookLM · ChatGPT** 모두 같은 한계.

**③ 중앙 — 비교 표 (좌·우 대칭, *칸 높이 컴팩트* — 한 줄씩만)**

| 현재 방식 (RAG · NotebookLM · ChatGPT) | LLM Wiki |
|---|---|
| 매 질문마다 raw 재검색·재조립 | 한 번 컴파일된 지식이 **축적** |
| 답변은 채팅 로그로 **휘발** | 좋은 답변은 위키에 **다시 적립** |
| 5문서 종합 → 매번 처음부터 | 교차 참조·모순 표시가 **이미 되어 있음** |
| 시간이 지나도 **그대로** | 시간이 지날수록 **풍부해짐** |

> *디자인 노트:* 셀 padding 최소, 글자 크기 조금 줄여서 표 전체 높이 압축 — 아래 답변 예시 블록까지 한 화면에 들어가게.

**④ 표 아래 — 살아있는 답변 예시 (수직 두 블록, 부피 비대칭)**

> **Q. *"삼성전자 내년 목표가는?"***
>
> **▶ 단순 요약 (RAG / NotebookLM / ChatGPT) 의 답**
> &nbsp;&nbsp;&nbsp;*"GS 리포트(4/22)에 따르면 목표가 95,000원으로 상향."*
> &nbsp;&nbsp;&nbsp;→ 한 보고서 인용으로 끝. 다음 질문에 또 처음부터.

> **▶ LLM Wiki 의 답**
> &nbsp;&nbsp;&nbsp;· **컨센서스 95K** (GS 95 / MS 92 / JPM 98) — 3개월 전 평균 88K 대비 **+7K 상향**

  ③ ✍️  노트에 손으로 정리                ↔    ③ 📄  .md 파일로 자동 정리

## 슬라이드 5 — 일회성으로 매번 답을 찾는 LLM에서, 지식을 스스로 쌓고 진화하는 LLM으로

### 화면 구성 (위 → 아래)

**① 헤드라인 (큰 글씨, 두 줄 단정형)**

> **일회성으로 매번 답을 찾는 LLM에서,**
> **지식을 스스로 쌓고 진화하는 LLM으로.**

**② 중앙 본체 — 데이터 파이프라인 다이어그램**

```
                  ┌──────────────────────  Wiki = Codebase  ──────────────────────┐
                  │                                                                │
   INPUT          │    raw/      ─── 정리 ──▶    wiki/    ◀── 따른다 ── schema/   │      OUTPUT
   ─────          │   (원본)                  (정리된 지식)             (CLAUDE.md │      ──────
   PDF            │     ▲                        │  ▲                  운영 규칙) │      답변
   메일   ───────▶│     │                        ▼  │                             │ ────▶ 문서
   메모    ingest │     │                        lint                             │ query 보고
                  │     │                       (자정화)                           │
                  │     └────────  LLM (Programmer) — 모든 화살표 실행  ──────────┘
                  │                                                                │
                  │                              ▲                                 │
                  │              Obsidian (IDE) — wiki/ 를 보는 화면               │
                  └────────────────────────────────────────────────────────────────┘
```

**③ 명령 셋 풀이 (다이어그램 직하, 한 줄씩)**

- **ingest** — 외부 자료 → raw → 정리하여 wiki에 반영
- **query** — wiki를 읽어 답변·문서 생성
- **lint** — wiki 자체를 주기적으로 점검 (모순 · 누락 · 오래됨)

**④ 본부 적용 라벨 (다이어그램 위·아래에 회색 작은 글씨로 부착)**

| 위치 | 본부 라벨 |
|---|---|
| INPUT 아래 | *"셀사이드 PDF · 사내 메모 · IR 자료 · 매크로 노트"* |
| `raw/` 아래 | *"raw/2026-04/GS_삼성전자.pdf"* |
| `wiki/` 아래 | *"wiki/종목/삼성전자.md, wiki/섹터/메모리.md"* |
| `schema/` 아래 | *"본부 톤 · 강조 기준 · 컴플라이언스 한 번 박음"* |
| OUTPUT 아래 | *"운용역 query 답변 · 분기 보고서 자동 정리"* |

**⑤ 핵심 인용 (작게)**

> *"Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다."* — Karpathy

**⑥ 마무리 한 줄 (하단 가운데, 굵게)**

> 답이 채팅에 *휘발* 이 아니라 **wiki에 누적** — 시간 지날수록 풍부.

> ***Wiki 박스 안을 들여다보면 세 종류의 데이터가 있습니다.** `raw/` 는 원본 그대로 보관, `wiki/` 는 정리된 지식 (종목 페이지, 섹터 페이지 등), `schema/` 는 운영 규칙 — *본부 톤, 강조 기준, 컴플라이언스* 를 사람이 한 번 박아두면 LLM이 따릅니다.*
>
> ***데이터를 움직이는 건 LLM 입니다 — 카파시 비유로 *프로그래머*.** 세 가지 일을 합니다. 자료가 들어오면 정리해서 wiki에 반영 (*ingest*). 질문에 답할 때 wiki를 읽고 출력 (*query*). 그리고 주기적으로 wiki 자체를 점검 — 모순·누락·오래된 내용 (*lint*). 이게 *자정화* 입니다.*
>
> *사람은 어디 있나요? **Obsidian (IDE) 으로 wiki를 봅니다.** 그리고 schema (CLAUDE.md) 를 한 번 박아두는 게 사람의 가장 중요한 작업입니다. 일상 운영은 LLM이 합니다.*
>
> ***이게 왜 중요한가** — 답이 채팅 로그로 휘발하지 않고 wiki에 누적됩니다. 다음 사람의 query는 더 빠르고 더 풍부한 답으로 돌아옵니다. **시간이 지날수록 본부 자료가 자라는 구조입니다.***
>
> *그러면 *누가 어떻게* 이걸 시작하는가 — 마지막 장에서."*

|---|---|---|---|
| ① **HEADLINE** | 상단 띠 | 전체 | *비유 선언* — Karpathy 원문 직접 인용 |
| ② **HOW** (PM 4가지 일) | 가운데 좌 | 60% | *행동* — 본체, 시선이 가장 먼저 닿음 |
| ③ **WHY** (3세대 박스) | 가운데 우 | 40% | *프레이밍* — 행동의 의미 풀이 |
| ④ **강조 + 인용** | 하단 띠 | 전체 | *행동 명령 + 권위 마무리* |

### 디자인 노트

- **좌·우 분할 비율 60:40** — PM 4가지 일이 본체이므로 좌측이 더 넓음. 3세대 박스는 보조 프레이밍.
- **HOW vs WHY 영어 라벨** — 임원이 시각적으로 *"이쪽은 행동, 저쪽은 의미"* 즉시 인지. 한국어 *"방법 / 이유"* 보다 짧고 명료. 라벨 자체를 빼도 영역 분할만으로 인지 가능 — 시안 후 결정.
- **HOW와 WHY가 *평행* 배치** — 임원의 시선 동선: 좌측 4가지 행동 훑기 → 우측 *"이게 결국 3세대"* 결론 즉시 인지. 세로 7항목 나열이면 *"4가지 → 한참 후 → 3세대"* 흐름이라 결론 도달 늦음.
- **PM 사이클 다이어그램 제거** — v4·v5 회색 보조였으나 PM 4가지 일이 사이클을 이미 풀이. 시각적 중복 제거로 4영역 분할이 깔끔해짐. 발표자 멘트 한 줄로만 흡수: *"나(PM)가 디렉션·자료를 주고, LLM이 .md 산출, Wiki가 자란다. 검토는 다시 PM."*
- **본부장 자리 박스 부재** — v5_page6 결정 그대로 (사용자 명시 삭제).
- **밑줄·강조 활용:** ① 헤드라인은 한 줄 굵게, ④ 강조는 양쪽 ★ 또는 굵은 박스로 시각 강조. ②③ 본체는 글머리 기호로 정렬.

### 발표자 멘트

> *"마지막입니다. 지금까지 보신 그림에 한 조각만 더 얹겠습니다.*
>
> ***앞 장에서 LLM이 정리·답변·점검을 한다고 했습니다. 그러면 사람은 뭘 하나** — 두 가지가 바뀝니다. **애널리스트는 *수집·요약·분류* 라는 반복 작업에서 풀려 *분석·판단·소통* 에 집중합니다. 운용역은 *'그 리포트 누가 봤더라?'* 검색에서 풀려 *질문을 던지는 사람* 이 됩니다.** 위키가 애널리스트를 *대체* 하는 게 아니라, 애널리스트가 더 위로 올라갑니다.*
>
> *카파시는 이걸 한 줄로 정리했습니다 — **'Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다.'** 그러면 한 자리가 비어 있습니다. 그 코드베이스를 가진 프로젝트의 **PM은 누구인가.** 그게 우리입니다.*
>
> ***화면을 보시면 왼쪽이 행동, 오른쪽이 그 행동의 의미입니다.** 왼쪽부터 — PM이 하는 일은 네 가지입니다. 첫째, **무엇을 담을지 정합니다** — 내가 커버하는 종목, 매크로 노트, 모델 기록. **앞 장 wiki/ 폴더 분류(종목·섹터·매크로)가 바로 이 결정의 결과입니다.** 둘째, **자료를 던져줍니다** — 매일 받는 PDF·메모·메일을 폴더에 떨굽니다. 셋째, **방향을 지속적으로 피드백합니다** — `schema/CLAUDE.md` 한 장에 *어떤 톤으로, 무엇을 강조해서 정리할지* 박아두고, 결과 보면서 다듬습니다. 넷째, **질문하고 검토합니다** — 지난 분기 의견 흐름, 모순 점검. **여기서 *검토* 는 사람의 판단입니다.** 위키가 *정리* 까지, *판단* 은 사람.*
>
> ***이 네 가지 일이 결국 무엇인지 — 오른쪽을 보시면 됩니다. 3세대 문제 해결 방법입니다.***
>
> *1세대는 전통적인 방식 — *사람이 규칙을 명시적인 코드로* 작성했습니다. `if A then B`. 2세대는 머신러닝 — *데이터로부터 규칙을 모델 가중치로* 학습했습니다. **그리고 지금 — 3세대는 더 이상 코드를 작성하지도, 모델을 학습하지도 않습니다. LLM에게 *자연어로* 지시합니다.***
>
> ***그러면 우리가 만드는 Wiki는 무엇인가** — *우리의 경험·인사이트·직관이 녹은 코드* 입니다. 본부의 지식이 곧 본부의 *코드베이스* 가 됩니다. 카파시가 *"Wiki는 코드베이스다"* 라고 한 건 단순 비유가 아니라 본질입니다. **종목 페이지·섹터 페이지·schema/CLAUDE.md — 이 모든 .md가 본부 일을 자동으로 풀어주는 코드입니다.***
>
> *그래서 — **코딩하지 않습니다.** 카파시가 *바이브코딩* 이라는 말을 만든 사람인데, 본질은 *'코드 못 짜도 도메인 알면 만들 수 있다'* 입니다. 위키도 똑같습니다. **본부에 계신 분들은 이미 도메인 전문가입니다. 그게 PM 자격입니다.***
>
> *시작에 필요한 건 폴더 하나입니다. 본부 차원 합의도, 예산도, 결정도 필요 없습니다. 각자 자기 영역에서, 오늘부터 시작하실 수 있습니다.*
>
> ***카파시 본인이 말한 대로 — 새로운 종류의 제품이 가능해진 시점입니다.** 본부 안에서도 그 시점이 시작되었으면 좋겠다는 게 오늘 발표의 결론입니다."*

**채택 이유:** 임원 PPT 발표에는 좌·우 *행동(HOW) vs 의미(WHY)* 가 한 화면에 동시에 보여서 청중이 *"4가지 일이 결국 패러다임 전환"* 결론을 즉시 흡수. 세로 7항목 형태는 *문서 읽기* 에는 좋지만 *5분 발표 슬라이드* 에는 빽빽함.
