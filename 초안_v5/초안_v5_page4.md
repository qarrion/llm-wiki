## 슬라이드 4 — 핵심 아이디어 : 단순 요약 → 추출 · 구조화 · 누적

### 화면 구성 (위 → 아래)

**① 헤드라인 (큰 글씨)**

> **핵심 아이디어 :** 단순 요약 → 추출 · 구조화 · 누적.

**② 중앙 다이어그램 — 본체 3박스 (큰 글씨, 굵은 테두리)**

```
   Obsidian                LLM 에이전트                Wiki
  (보기 화면)               (정리하는 손)             (지식의 집)
   ┌──────────┐            ┌────────────┐         ┌──────────────┐
   │ 사람이    │ ◀── view ─ │  Ingest    │ ─ write ▶  주제별 .md   │
   │ 보는 화면 │            │  Query     │          │  요약·관계   │
   │          │            │  Lint      │          │  개념·비교   │
   └──────────┘            └────────────┘         └──────────────┘
```

**③ 운영 사이클 (3박스 아래, 한 줄씩)**

- **Ingest** — 새 자료가 들어오면 관련 위키 페이지 자동 갱신.
- **Query** — 질문에 답하고, 좋은 답은 다시 위키로 적립.
- **Lint** — 주기적으로 모순·누락·오래된 내용을 점검.

**④ 보조 (회색 작은 영역, 화면 하단)**

```
부속:  raw/ (원본 자료, LLM이 읽기만)
       index.md (색인·작업로그, LLM이 자동 유지)
       CLAUDE.md (운영 규칙, 사람이 한 번만 박아둠)
```

**⑤ 핵심 인용 (다이어그램 직하, 강조)**

> *"Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다."* — Karpathy

**⑥ 마무리 한 줄 (하단 가운데, 회색)**

> 문서가 아니라, **다시 쓸 수 있는 자산** 으로.

### 발표자 멘트

> *"앞 슬라이드의 답답함을 어떻게 푸는가 — 이 그림이 오늘 발표의 본체입니다.*
>
> *세 박스만 보시면 됩니다. **왼쪽이 Obsidian — 사람이 보는 화면. 가운데가 LLM 에이전트 — 정리하는 손. 오른쪽이 Wiki — 지식의 집. 주제별 마크다운 파일들의 모음입니다.** LLM이 위키를 직접 작성하고 유지합니다. 사람은 결과만 봅니다.*
>
> *LLM이 하는 일은 세 가지입니다. 새 자료가 들어오면 흡수해서 관련 페이지를 갱신하고 — *Ingest*. 질문에 답하면서 그 답을 다시 위키로 쌓고 — *Query*. 주기적으로 모순이나 누락을 점검합니다 — *Lint*. 이 셋이 매일 돌아갑니다.*
>
> ***앞 장에서 본 답변 한 쌍** 을 떠올려 주시면 — 좌측 단순 요약은 *raw 한 보고서* 에서 직접 답했고, 우측 LLM Wiki 답은 *주제별 .md(컨센서스 / HBM / 환율 가정)* 들이 이미 정리되어 있어서 그 *관계* 에서 답이 만들어졌습니다. **그 .md 파일들의 집이 바로 우측 Wiki 박스입니다.** *컴파일된 지식* 이라는 표현이 그 의미였습니다.*
>
> *카파시는 이걸 이렇게 표현했습니다 — **'Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다.'** 이 비유가 오늘 발표 끝까지 따라갑니다.*
>
> *중요한 점 — **이게 별도 서버나 RAG 인프라가 필요한 구조가 아닙니다.** 마크다운 파일 폴더 하나로 시작합니다. 그래서 파일럿 비용이 거의 0입니다."*

### 임원 Q&A 대비

| 예상 질문 | 답 |
|---|---|
| *"위키를 LLM이 다 쓴다고? 사람이 검토 안 하나?"* | "검토 단계를 넣을 수 있습니다. 첫 1~2주는 모든 결과를 사람이 확인하는 모드로 시작하면 됩니다. *Lint* 가 모순을 자동으로 드러내기 때문에 검토 부담도 시간이 갈수록 줄어듭니다." |
| *"Obsidian을 꼭 써야 하나? 우리는 Notion인데."* | "Obsidian은 *보기 화면* 일 뿐 본질이 아닙니다. 본체는 마크다운 파일이고, Notion에 import하는 방식도 가능합니다. 도구 종속성 없습니다." |
| *"이게 NotebookLM이랑 뭐가 다른가?"* | "NotebookLM은 *raw 문서만 보고 매번 처음부터* 답합니다. 여기는 raw와 답 사이에 *유지되는 위키* 가 있고, 좋은 답이 *다시 그 위키에 쌓입니다*. 슬라이드 3의 답답함이 풀리는 자리가 바로 이 차이입니다." |
| *"문서가 100개 넘어가면 LLM이 다 못 읽지 않나?"* | "카파시 본인이 *~100개 문서·40만 단어* 까지 별도 인프라 없이 작동한다고 명시했습니다. 본부 첫 파일럿 규모는 이 안에 들어옵니다 — *색인 + 요약* 을 LLM이 자동으로 유지하기 때문입니다." |
| *"기존 RAG 시스템 도입과 뭐가 다른가?"* | "RAG는 *질문할 때마다 raw에서 chunk를 가져옴* — 정리는 안 됨. 위키는 *정리된 .md를 직접 읽음* — chunk 단위가 아닌 *정리된 지식 단위*. 답변 품질 + 추적 가능성 모두 다릅니다." |
| *"Lint가 모순을 자동으로 표시한다고 했는데, LLM이 판단도 하나?"* | "**아닙니다.** Lint는 *모순이 있다* 는 사실까지만 표시 — *어느 쪽이 맞는지* 결정은 사람 몫입니다. *Query* 답변 적립도 마찬가지 — *좋은 답* 인지는 사람이 검토하고 위키에 들어갑니다. 위키가 *판단* 하는 게 아니라 *판단의 근거를 정리* 합니다." |

### 슬라이드 5와의 연결

다이어그램의 *Ingest · Query · Lint 세 동작* 이 슬라이드 5에서 **본부 일에서 가능해지는 4가지** 로 풀린다.

**다리 한 줄 (발표자 멘트 마지막):**

> *"그러면 이 구조가 본부 일에서 *구체적으로* 뭘 가능하게 하는가 — 다음 장입니다."*

→ 슬라이드 5 헤드라인 *"그래서, 우리가 뭘 할 수 있게 됐는데?"* 으로 자동 진입.

청중 머리 속 흐름:
1. *(슬라이드 3)* "차이 알겠다 — 비교표 + 답변 예시. 근데 그 차이를 만드는 *구조* 는?"
2. *(슬라이드 4 다이어그램)* "아, Obsidian / LLM 에이전트 / Wiki 세 박스."
3. *(운영 사이클)* "Ingest, Query, Lint 셋이 돈다."
4. *(page3 답변 회수 한 단락)* "**앞 장에서 본 답변 우측이 바로 이 Wiki에서 나온 거구나.**"
5. *(인용)* "Wiki = 코드베이스. 좋은 비유."
6. *(자동 의문)* "근데 본부 일에 정확히 뭘 해주는 건데?"
7. *(슬라이드 5)* "네 가지 — 그 중 *질문이 자산이 된다* 가 핵심."
