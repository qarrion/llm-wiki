## 슬라이드 4a — 핵심 아이디어 : 사람이 하던 정리, LLM이 그대로

### 화면 구성 (좌·우 평행 비교)

**① 헤드라인 (큰 글씨)**

> **핵심 아이디어 :** 사람이 하던 정리, LLM이 그대로 — 0 비용으로.

**② 중앙 — 좌·우 평행 다이어그램 (5단계 1:1 대응, 이미지 + 화살표)**

```
       [ 수기 — 사람이 직접 ]                    [ 자동 — Claude Code + Obsidian ]

  ① 📰  자료를 본다                       ↔    ① 🌐  자료가 들어온다
       (책 · 신문 · 회의 노트)                       (인터넷 · 사내 PDF · 셀사이드 메일)
            │                                              │
            ▼                                              ▼
  ② 🧠  생각하고 해석한다                  ↔    ② 🤖  Claude Code (LLM) 가 추출
       ("이게 무슨 의미지?")                          ("이 자료에서 뭘 정리하지?")
            │                                              │
            ▼                                              ▼
  ③ ✍️  노트에 손으로 정리                ↔    ③ 📄  .md 파일로 자동 정리
            │                                              │
            ▼                                              ▼
  ④ 📁  폴더·캐비넷에 분류 저장            ↔    ④ 🗂  Obsidian vault에 자동 저장
            │                                              │
            ▼                                              ▼
  ⑤ 📚  나중에 꺼내본다                   ↔    ⑤ 🔍  나중에 query · 그래프 뷰
       (찾기 어려움 / 휘발 / 정리비용 ↑)              (즉시 검색 / 영구 / 정리비용 0)
```

> *디자인 노트:*
> - 좌·우 모두 5단계, 같은 높이로 평행 배치 — 단계마다 좌·우 사이 ↔ 화살표가 *"같은 일이다"* 시각적 강조.
> - 좌측은 *수기 사진·실물 아이콘*, 우측은 *스크린샷·디지털 아이콘* 으로 매체 대비.
> - **⑤번 우측의 그래프 뷰는 슬라이드 1에서 본 그 그림** — page1 회수.

**③ 핵심 인용 (다이어그램 직하, 작게)**

> *"Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다."* — Karpathy

**④ 마무리 한 줄 (하단 가운데, 굵게)**

> **사람이 평생 한 번 하던 정리를, LLM은 매일 한다.**

### 발표자 멘트

> *"앞 장의 답답함을 어떻게 푸는가 — 이 그림 한 장으로 보여드리겠습니다.*
>
> *왼쪽이 우리가 평소에 하던 *수기* 정리 흐름입니다. ① 자료를 본다 → ② 머리로 해석한다 → ③ 노트에 적는다 → ④ 폴더에 넣는다 → ⑤ 나중에 꺼내본다. **다섯 단계 모두 사람이 직접 합니다.** 그래서 시간이 지나면 정리비용이 쌓이고, 휘발되고, 찾기 어려워집니다.*
>
> *오른쪽이 LLM이 같은 일을 하는 흐름입니다. **단계가 똑같습니다.** ① 인터넷·PDF가 들어오면 → ② Claude Code 같은 LLM 에이전트가 읽고 추출 → ③ 마크다운 파일로 정리 → ④ Obsidian vault 에 자동 저장 → ⑤ 나중에 query 또는 그래프 뷰로 꺼내봅니다. **다섯 단계 모두 LLM이 자동으로 합니다.***
>
> ***⑤번 우측의 그래프 뷰가 — 슬라이드 1에서 본 그 그림입니다.** 수기로 평생 모은 노트의 디지털 결과물이 저 그림인 셈입니다.*
>
> *카파시는 이걸 이렇게 표현했습니다 — **'Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다.'** 사람이 평생 한 번 하던 정리를, LLM은 매일 합니다. 그것도 별도 인프라 없이, 마크다운 폴더 하나에서."*

### 임원 Q&A 대비

| 예상 질문 | 답 |
|---|---|
| *"사람 정리랑 LLM 정리, 품질이 같나?"* | "다릅니다 — 수기는 *깊이*, LLM은 *양·일관성·연결성*. 둘이 *경쟁* 아니라 *합쳐짐* — LLM이 양·연결을 깔고, 사람이 그 위에 깊이를 더합니다. ⑤번에서 사람이 query 하는 순간 두 정리가 합쳐집니다." |
| *"LLM이 잘못 정리하면?"* | "수정합니다. LLM이 만든 .md는 *그대로 마크다운* 이라 사람이 직접 열어 고칠 수 있습니다. 별도로 *모순 자동 점검* (Lint) 도 돌아 — 잘못이 *감춰지지 않고 드러나는* 구조입니다." |
| *"Obsidian이 핵심인가? 우리는 Notion인데."* | "Obsidian은 *보기 화면* 일 뿐 본질이 아닙니다. 본체는 마크다운 파일이고 — Notion으로 import 가능, VS Code로도 열림. 도구 종속성 없습니다." |
| *"Claude Code 라는 건 뭔가? 또 새 도구?"* | "Cursor / Claude Code / ChatGPT 등 LLM 에이전트 도구 중 어느 것이든 가능. 핵심은 *LLM이 마크다운 파일을 직접 작성·수정* 한다는 패턴. 도구는 부차적입니다." |
| *"사람이 평생 한 정리를 LLM이 매일 한다 — 그럼 사람은 뭘 하나?"* | "**④ 분류 와 ⑤ 꺼내보기 에서 풀려나, ② 해석 과 *질문* 에 집중합니다.** 슬라이드 5에서 그 변화를 보여드립니다 — *반복 정리에서 분석·판단으로*." |
| *"수기랑 자동 둘 다 같이 가나?"* | "그렇습니다. 좌측 수기 노트(메모·필기)도 우측 vault에 *함께* 들어갑니다 — `raw/` 폴더에 사진·스캔으로. LLM이 OCR 후 정리. 둘이 *분리* 가 아니라 *합류* 입니다." |
| *"LLM이 정리·저장까지 한다는데, 그럼 판단도 하나?"* (= 컴플라이언스) | "**아닙니다.** ②~④ (추출·정리·저장) 까지는 LLM이 자동, ⑤ *꺼내볼 때 판단* 은 사람 몫. 위키는 *판단의 근거를 정리* 하지 *판단 자체* 는 안 합니다." |

### 슬라이드 5와의 연결

다이어그램의 *5단계 평행 흐름* 이 슬라이드 5에서 **본부 일에서 가능해지는 4가지** 로 풀린다.

**다리 한 줄 (발표자 멘트 마지막):**

> *"그러면 이 흐름이 본부 일에서 *구체적으로* 뭘 가능하게 하는가 — 다음 장입니다."*

→ 슬라이드 5 헤드라인 *"그래서, 우리가 뭘 할 수 있게 됐는데?"* 으로 자동 진입.

청중 머리 속 흐름:
1. *(슬라이드 3)* "차이 알겠다 — 단순 요약 vs LLM Wiki. 근데 LLM이 정확히 어떻게 정리하는데?"
2. *(슬라이드 4a 좌측)* "아 — 사람이 평소 하던 흐름. 익숙하다."
3. *(슬라이드 4a 우측)* "**LLM이 같은 5단계를 그대로 하네.** 단계마다 1:1 대응."
4. *(⑤번 우측)* "그래프 뷰는 슬라이드 1에서 본 거. 회수."
5. *(인용)* "Obsidian = IDE, LLM = 프로그래머, Wiki = 코드베이스. 좋은 비유."
6. *(자동 의문)* "근데 본부 일에 정확히 뭘 해주는 건데?"
7. *(슬라이드 5)* "네 가지."
