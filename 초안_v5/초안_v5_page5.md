## 슬라이드 5 — 일회성으로 매번 답을 찾는 LLM에서, 지식을 스스로 쌓고 진화하는 LLM으로

### 화면 구성 (위 → 아래)

**① 헤드라인 (큰 글씨, 두 줄 단정형)**

> **일회성으로 매번 답을 찾는 LLM에서,**
> **지식을 스스로 쌓고 진화하는 LLM으로.**

**② 중앙 본체 — 데이터 파이프라인 다이어그램**

```
                  ┌──────────────────────  Wiki = Codebase  ──────────────────────┐
                  │                                                                │
   INPUT          │    raw/      ─── 정리 ──▶    wiki/    ◀── 따른다 ── schema/   │      OUTPUT
   ─────          │   (원본)                  (정리된 지식)             (CLAUDE.md │      ──────
   PDF            │     ▲                        │  ▲                  운영 규칙) │      답변
   메일   ───────▶│     │                        ▼  │                             │ ────▶ 문서
   메모    ingest │     │                        lint                             │ query 보고
                  │     │                       (자정화)                           │
                  │     └────────  LLM (Programmer) — 모든 화살표 실행  ──────────┘
                  │                                                                │
                  │                              ▲                                 │
                  │              Obsidian (IDE) — wiki/ 를 보는 화면               │
                  └────────────────────────────────────────────────────────────────┘
```

**②-2 `wiki/` 안을 들여다보면 — zoom-in 박스 (다이어그램 우측 또는 직하)**

```
wiki/   ← 주제별로 요약 .md가 만들어진다.

  종목/
    삼성전자.md       ← 한 종목의 모든 의견·변화·연결이 한 페이지
    SK하이닉스.md

  섹터/
    메모리.md         ← 섹터 차원의 컨센서스·이슈·종목 비교

  매크로/
    USDKRW.md         ← 매크로 변수 하나의 흐름과 전망 정리

  index.md            ← 전체 색인 (LLM 자동 유지)
  log.md              ← 작업 로그 (감사·역추적)
```

> *주제별 .md 한 장 = 본부 차원의 한 페이지 위키*

> *디자인 노트:*
> - 좌측 첫 줄(*"주제별로 요약 .md가 만들어진다"*)이 **추상 형태**, 그 아래 트리가 **본부 적용 예시**. 한 박스에서 *"왜 이렇게 정리하나"* + *"내 일에 어떻게 들어오나"* 동시 답.
> - 시각 배치는 ②다이어그램 *우측* 에 두면 좌·우 분할로 시선 동선 자연스러움. 슬라이드 도구에서 가로 1:1 또는 3:2 분할 권장.

**③ 명령 셋 풀이 (다이어그램 직하, 한 줄씩)**

- **ingest** — 외부 자료 → raw → 정리하여 wiki에 반영
- **query** — wiki를 읽어 답변·문서 생성
- **lint** — wiki 자체를 주기적으로 점검 (모순 · 누락 · 오래됨)

**④ 본부 적용 라벨 — 단순화 (INPUT·OUTPUT 만)**

| 위치 | 본부 라벨 |
|---|---|
| INPUT 아래 | *"셀사이드 PDF · 사내 메모 · IR 자료 · 매크로 노트"* |
| OUTPUT 아래 | *"운용역 query 답변 · 분기 보고서 자동 정리"* |

> *raw/ · wiki/ · schema/ 의 본부 적용 예시는 ②-2 zoom-in 박스에서 보여줌.*

**⑤ 핵심 인용 (작게)**

> *"Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다."* — Karpathy

**⑥ 마무리 한 줄 (하단 가운데, 굵게)**

> 답이 채팅에 *휘발* 이 아니라 **wiki에 누적** — 시간 지날수록 풍부.

### 발표자 멘트

> *"앞 장의 답답함을 어떻게 푸는가 — 이 한 그림이 오늘 발표의 본체입니다.*
>
> ***좌→우 한 줄로 보시면 됩니다.** 왼쪽 INPUT — 본부에 매일 들어오는 셀사이드 PDF · 메모 · IR 자료. 가운데 박스가 *Wiki* 전체 — 카파시 표현으로 *코드베이스* 입니다. 오른쪽 OUTPUT — 운용역이 받는 답변, 보고서.*
>
> ***Wiki 박스 안을 들여다보면 세 종류의 데이터가 있습니다.** `raw/` 는 원본 그대로 보관, `wiki/` 는 정리된 지식, `schema/` 는 운영 규칙 — *본부 톤, 강조 기준, 컴플라이언스* 를 사람이 한 번 박아두면 LLM이 따릅니다.*
>
> ***그 `wiki/` 를 다시 한 번 펼쳐 보면** — 우측 zoom-in 박스입니다. ***주제별로 요약 .md가 만들어집니다.*** 종목별로 *삼성전자.md, SK하이닉스.md* — 한 종목의 모든 의견·변화·연결이 한 페이지에 정리. 섹터별로 *메모리.md, 디스플레이.md* — 섹터 컨센서스와 종목 비교. 매크로별로 *USDKRW.md, 금리.md*. 그리고 **`index.md`** 가 전체 색인을 LLM이 자동 유지, **`log.md`** 가 작업 로그·감사 흔적입니다. **사람이 본부 폴더 구조를 짜듯 LLM도 그 구조 위에서 정리합니다.***
>
> ***데이터를 움직이는 건 LLM 입니다 — 카파시 비유로 *프로그래머*.** 세 가지 일을 합니다. 자료가 들어오면 정리해서 wiki에 반영 (*ingest*). 질문에 답할 때 wiki를 읽고 출력 (*query*). 그리고 주기적으로 wiki 자체를 점검 — 모순·누락·오래된 내용 (*lint*). 이게 *자정화* 입니다.*
>
> *사람은 어디 있나요? **Obsidian (IDE) 으로 wiki를 봅니다.** 그리고 schema (CLAUDE.md) 를 한 번 박아두는 게 사람의 가장 중요한 작업입니다. 일상 운영은 LLM이 합니다.*
>
> ***이게 왜 중요한가** — 답이 채팅 로그로 휘발하지 않고 wiki에 누적됩니다. 다음 사람의 query는 더 빠르고 더 풍부한 답으로 돌아옵니다. **시간이 지날수록 본부 자료가 자라는 구조입니다.***
>
> *그러면 *누가 어떻게* 이걸 시작하는가 — 마지막 장에서."*

### 임원 Q&A 대비

| 예상 질문 | 답 |
|---|---|
| *"raw, wiki, schema 차이가 뭔가?"* | "*raw* 는 손대지 않은 원본 (감사·역추적용). *wiki* 는 LLM이 정리한 지식 (종목·섹터 페이지). *schema* 는 사람이 박아둔 운영 규칙 (`CLAUDE.md`). 셋이 분리되어야 *원본 추적* 과 *정리된 답* 이 모두 가능합니다." |
| *"ingest/query/lint 가 다 자동인가? 사람은 안 끼나?"* | "기본은 자동. 다만 *Lint* 가 모순을 표시하면 *어느 쪽이 맞는지* 판단은 사람. *Query* 답변도 *위키로 적립* 단계에서 사람 검토 모드를 둘 수 있습니다." |
| *"schema (CLAUDE.md) 는 누가 어떻게 쓰나?"* | "**한 번만 사람이 박습니다.** *우리 본부는 어떤 톤으로, 무엇을 강조해서 정리한다* 한 페이지면 충분. 한 번 박으면 LLM이 따릅니다. 슬라이드 6 PM의 일 ④번이 바로 이 작업." |
| *"wiki/ 폴더 분류(종목/섹터/매크로)는 누가 정하나?"* | "**처음 한 번 사람이 정합니다** — 본부 운용 분류 그대로. 새 종목·섹터가 추가되면 LLM이 schema 규칙에 따라 적절한 폴더에 자동 배치. 분류 자체를 바꾸려면 사람이 schema 한 줄 수정." |
| *"index.md, log.md는 누가 만드나?"* | "**LLM이 자동 유지** 합니다. 새 .md가 추가되면 색인이 갱신되고, 모든 작업이 log에 기록 — 감사·역추적용. 사람이 손댈 일 없습니다." |
| *"Obsidian이 핵심인가? 우리는 Notion인데."* | "Obsidian은 *보기 화면(IDE)* 일 뿐 본질이 아닙니다. 본체는 마크다운 파일이고 Notion으로도 import 가능. 도구 종속성 없습니다." |
| *"기존 RAG 시스템 도입과 뭐가 다른가?"* | "RAG는 *질문할 때마다 raw에서 chunk를 가져옴* — 정리는 안 됨. LLM-WIKI는 *정리된 wiki를 직접 읽음* — 답변 품질 + 추적 가능성 모두 다릅니다. 슬라이드 3 비교표 ②③행이 그 차이." |
| *"위키가 *판단* 도 하나? (종목 추천 같은)"* | "**아닙니다.** 위키는 정리·관계까지 — 결론(*'사야 한다'*)은 만들지 않습니다. *Lint* 도 모순 *표시* 까지, *어느 쪽이 맞는지* 는 사람. 슬라이드 6에서 그 선이 한 번 더 명시됩니다." |
| *"문서가 100개 넘으면 LLM이 다 못 읽지 않나?"* | "카파시 본인이 *~100개 문서·40만 단어* 까지 별도 인프라 없이 작동한다고 명시. 본부 첫 파일럿 규모는 이 안에 들어옵니다 — `index.md` (색인) 를 LLM이 자동 유지." |

### 슬라이드 6과의 연결

다이어그램의 *데이터 흐름* 과 *명령 셋* 이 슬라이드 6에서 **PM이 하는 4가지 일** 로 자연스럽게 받아진다.

**다리 한 줄 (발표자 멘트 마지막):**

> *"그러면 *누가 어떻게* 이걸 시작하는가 — 마지막 장에서."*

→ 슬라이드 6 헤드라인 *"이제, 우리가 PM입니다"* 으로 자동 진입.

청중 머리 속 흐름:
1. *(슬라이드 3)* "차이 알겠다 — 단순 요약 vs LLM Wiki. 근데 LLM이 정확히 어떤 구조에서 그 차이를 만드는데?"
2. *(슬라이드 5 다이어그램 좌→우)* "아 — 데이터가 raw로 들어와서, wiki로 정리되고, query로 답이 나간다. 한 줄 흐름."
3. *(raw/wiki/schema 3구조)* "원본·정리·규칙 분리. raw는 추적용, wiki가 본체, schema는 사람이 한 번 박는 자리."
4. *(②-2 zoom-in)* "**wiki/ 안에 종목/섹터/매크로 폴더로 .md 가 정리되는구나.** 본부 분류 그대로."
5. *(ingest/query/lint 명령)* "LLM이 자동으로 도는 세 동작."
6. *(역할 매핑)* "Obsidian = IDE, LLM = Programmer, Wiki = Codebase. 카파시 비유 명확."
7. *(본부 라벨)* "셀사이드 PDF가 INPUT, 운용역 답변·보고서가 OUTPUT — 본부 일에 직접 매핑됨."
8. *(마무리)* "답이 wiki에 누적 → 시간 지나면 풍부. 본부 자산화."
9. *(자동 의문)* "그래서 누가 시작하는데?"
10. *(슬라이드 6)* "각자가 PM. 폴더 하나에서 오늘부터."

**왜 이 변경 (사용자 결정):**
- *원리가 주* 라는 사용자 명시 — 데이터 파이프라인 한 그림에 *역할/구조/명령/데이터 흐름* 4가지를 모두 담음.
- *wiki/ 내부를 한 단계 더 형상화* — zoom-in 박스로 *주제별 요약*(추상) + *본부 분류 예시*(구체) 를 한 자리에. 임원이 *"왜 이렇게 정리하나"* + *"내 일에 어떻게 들어오나"* 동시 답.
- 본부 효용을 회색 라벨로 강등 → 임원이 *원리를 먼저 이해하고, 그 위에 본부 적용을 자연스럽게 얹음*.
