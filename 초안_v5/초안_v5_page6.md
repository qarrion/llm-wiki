## 슬라이드 6 — LLM은 프로그래머. Wiki는 코드베이스.

### 화면 구성 (위 → 아래)

**① 헤드라인 (큰 글씨, 한 줄)**

> **LLM은 프로그래머. Wiki는 코드베이스.**

**② 부제 (작게, 한 줄)**

> 자연어를 통한 새로운 문제 해결 방식.

**③ 중앙 본체 — PM이 하는 4가지 (큰 글씨·중앙·주인공)**

```
①  무엇을 담을지 정한다
    "내 커버 종목" / "매크로 노트" / "모델 기록" / "분기 보고 자료" …
    → wiki/종목/  ·  wiki/매크로/  ·  wiki/모델/   (page5 zoom-in 분류 그대로)

②  자료를 던져준다
    매일 받는 PDF · 메모 · 메일을 폴더에 떨군다.
    → page5 다이어그램의 INPUT → raw/ 단계.

③  질문하고 검토한다
    "지난 분기 의견 흐름 정리" / "모순 점검" / "갱신된 페이지 요약"
    → 좋은 답은 위키로 적립, 잘못된 건 수정 지시.
    → 위키가 정리·관계까지,  **검토(판단) = 사람의 몫.**

④  한 번만 방향을 잡아둔다
    schema/CLAUDE.md 한 장 — 정리 방식 · 톤 · 강조 기준 · 컴플라이언스.
    → 한 번 박으면 LLM이 따른다.   (page5 schema/ 직접 회수)
```

**④ 우측 또는 하단 — PM 사이클 (회색·작게·보조)**

```
   나(PM) ─ 디렉션 · 자료 ─▶ LLM(프로그래머) ─ .md 산출 ─▶ Wiki(자란다)
     ▲                                                          │
     └──────────────────────  검토 · 다음 질문  ──────────────────┘
```

**⑤ 3세대 문제 해결 박스 (신규 — 본문 핵심 결론)**

```
┌────────────────────────────────────────────────────────────────┐
│                                                                │
│   결국, 이건  *3세대 문제 해결 방법*  입니다.                  │
│                                                                │
│     1세대 (전통):     사람이 규칙을  *명시적 코드로*  작성     │
│     2세대 (머신러닝):  데이터에서 규칙을  *모델 가중치로*  학습 │
│     3세대 (LLM-WIKI):  LLM에게  *자연어로*  지시               │
│                                                                │
│   →  Wiki = *자연어로 쓰인 프로그래밍 코드.*                   │
│      본부의 지식이 곧 본부의  *코드베이스*  가 된다.           │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

**⑥ 강조 (큰 글씨 한 줄, 가운데)**

> **각자 자기 자료를 한 폴더에. 거기서부터 위키가 자랍니다.**

**⑦ 마무리 인용 (우측 하단, 작게)**

> *"새로운 종류의 제품이 가능해졌다."* — Karpathy

### 발표자 멘트

> *"마지막입니다. 지금까지 보신 그림에 한 조각만 더 얹겠습니다.*
>
> ***앞 장에서 LLM이 정리·답변·점검을 한다고 했습니다. 그러면 사람은 뭘 하나** — 두 가지가 바뀝니다. **애널리스트는 *수집·요약·분류* 라는 반복 작업에서 풀려 *분석·판단·소통* 에 집중합니다. 운용역은 *'그 리포트 누가 봤더라?'* 검색에서 풀려 *질문을 던지는 사람* 이 됩니다.** 위키가 애널리스트를 *대체* 하는 게 아니라, 애널리스트가 더 위로 올라갑니다.*
>
> *카파시는 이걸 한 줄로 정리했습니다 — **'Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다.'** 그러면 한 자리가 비어 있습니다. 그 코드베이스를 가진 프로젝트의 **PM은 누구인가.** 그게 우리입니다.*
>
> *PM이 하는 일은 네 가지입니다. 첫째, **무엇을 담을지 정합니다** — 내가 커버하는 종목, 매크로 노트, 모델 기록. **앞 장 wiki/ 폴더 분류(종목·섹터·매크로)가 바로 이 결정의 결과입니다.** 둘째, **자료를 던져줍니다** — 매일 받는 PDF·메모·메일을 폴더에 떨굽니다. 셋째, **질문하고 검토합니다** — 지난 분기 의견 흐름, 모순 점검. **여기서 *검토* 는 사람의 판단입니다.** 위키가 *정리* 까지, *판단* 은 사람. 넷째, **한 번만 방향을 잡아둡니다** — `schema/CLAUDE.md` 한 장. *어떤 톤으로, 무엇을 강조해서 정리할지* 한 번 박아두면 LLM이 따릅니다.*
>
> ***이 네 가지 일이 결국 무엇인지 한 마디로 정리하면 — 3세대 문제 해결 방법입니다.***
>
> *1세대는 전통적인 방식 — *사람이 규칙을 명시적인 코드로* 작성했습니다. `if A then B`. 2세대는 머신러닝 — *데이터로부터 규칙을 모델 가중치로* 학습했습니다. **그리고 지금 — 3세대는 더 이상 코드를 작성하지도, 모델을 학습하지도 않습니다. LLM에게 *자연어로* 지시합니다.***
>
> ***그러면 우리가 만드는 Wiki는 무엇인가** — *자연어로 쓰인 프로그래밍 코드* 입니다. 본부의 지식이 곧 본부의 *코드베이스* 가 됩니다. 카파시가 *"Wiki는 코드베이스다"* 라고 한 건 단순 비유가 아니라 본질입니다. **종목 페이지·섹터 페이지·schema/CLAUDE.md — 이 모든 .md가 본부 일을 자동으로 풀어주는 코드입니다.***
>
> *그래서 — **코딩하지 않습니다.** 카파시가 *바이브코딩* 이라는 말을 만든 사람인데, 본질은 *'코드 못 짜도 도메인 알면 만들 수 있다'* 입니다. 위키도 똑같습니다. **본부에 계신 분들은 이미 도메인 전문가입니다. 그게 PM 자격입니다.***
>
> *시작에 필요한 건 폴더 하나입니다. 본부 차원 합의도, 예산도, 결정도 필요 없습니다. 각자 자기 영역에서, 오늘부터 시작하실 수 있습니다.*
>
> ***카파시 본인이 말한 대로 — 새로운 종류의 제품이 가능해진 시점입니다.** 본부 안에서도 그 시점이 시작되었으면 좋겠다는 게 오늘 발표의 결론입니다."*

→ 본부장: *"결정해줄 게 없는 자리"* 였음을 인지. **자발적 채택의 흐름.**
→ 임원·운용역: *"내 자리에서 시작 가능"* + *"이건 단순 도구 도입이 아니라 패러다임 변화"* 동시 인지.

**왜 이 변경:**
- v4 본체(PM 4가지 일)는 안정적이라 손대지 않음 (사용자 *"내용은 나쁘진 않아"* 확인).
- 헤드라인 *"이제, 우리가 PM입니다"* 는 *PM* 영어 약어가 임원에게 부담 → **Karpathy 비유 직접 인용** 으로 교체. 부제로 PM 표현 보존.

- 본부장 자리 박스 삭제 — 사용자 결정. 발표 전반에서 *"본부장에 부담 주지 않는다"* 메시지는 Q&A로만 처리.
