# 최종 v1 — 페이지별 예상 질문 / 답변

발표 중·후 청중이 던질 가능성이 있는 질문을 페이지 단위로 정리. 각 페이지의 *해당 슬라이드 본문 안 Q&A 표* 는 핵심 3-4개만 추리고, 이 문서는 *그 외에 떠오를 수 있는 질문* 까지 폭넓게 망라한다.

청중 유형은 3가지로 나눈다:

- **A. 도구·용어를 모르는 사람** (Claude / Obsidian / 마크다운 / 링크 등)
- **B. 임원 의사결정 회로** (도입/비용/보안/사내 자원 충돌)
- **C. 개발 조직 출신 송곳 질문** (대안 도구·검증·범용성)

## 페이지 1 — LLM-WIKI : LLM이 직접 자신만의 위키를 만든다

| 1A-3 | *"마크다운이 뭐예요?"* | "텍스트 파일로 문서를 쓰는 가장 단순한 형식입니다. 글머리·제목·링크를 기호 몇 개로 표시하는 정도. *LLM이 다루기 가장 편한 문서 형식* 이라 이번 방식이 마크다운 위에서 동작합니다." |
| 1A-4 | *"링크/백링크가 뭐길래 그게 그렇게 좋다는 건가요?"* | "문서 A에서 문서 B를 가리키면, 자동으로 B에서도 *'A가 나를 참조한다'* 라고 보여주는 기능입니다. 결과적으로 모든 문서가 그래프로 연결되고 — *어떤 노트가 어디서 인용됐는지* 가 일거에 보입니다. 21세기 지식 노동자가 *'내가 쓴 자료를 다시 찾기 어려운 불안감'* 을 해소하는 핵심 기능." |
| 1A-5 | *"오른쪽 그림(Knowledge Graph)에서 점 하나가 뭔가요? 사람? 부서? 문서?"* | "**노트 한 개 = 점 한 개** 입니다. 마크다운 파일 하나하나가 노드. 선은 *문서 사이의 링크* — 사람이 직접 만들 수도 있고, *오늘 핵심 메시지인* LLM이 자동으로 발견·연결할 수도 있습니다." |
| 1A-6 | *"저 그래프는 자동으로 만들어지는 건가요, 누가 그리는 건가요?"* | "Obsidian이 자동으로 그립니다. 사람·LLM이 만든 *링크 데이터* 를 시각화한 결과일 뿐 — 그래프 자체를 누가 그리는 게 아닙니다. *저 그림이 멋있어 보여서 만들자* 가 아니라 *문서를 쓰면 저 그림이 부산물로 따라온다* 는 점이 핵심." |

| 3A-1 | *"RAG가 뭐예요?"* | "*Retrieval-Augmented Generation* — *문서 검색* 으로 LLM 답을 보강하는 방식. 질문이 들어오면 ① 관련 문서 조각을 뽑고 ② 그 조각들을 LLM에게 보여줘 답을 만들게 함. *NotebookLM·ChatGPT 파일 업로드·사내 검색 챗봇* 모두 RAG의 변형. *매 질문마다 처음부터 검색* 한다는 점이 비교표 좌측 1행의 정체." |

| 3A-3 | *"`ai slop` 이 뭐예요?"* | "*AI 슬롭* — 생성형 AI가 만드는 *그럴듯하지만 깊이 없는 답·콘텐츠* 를 통칭하는 업계 비판어. *slop* = *남은 음식·구정물*. 인터넷에 양산된 무의미한 AI 글을 비꼬는 표현으로 정착. 단순 요약 답변이 *그 카테고리에 들어간다* 는 인식이 핵심." |
| 3A-4 | *"답변 예시의 `#메모리/HBM` 같은 태그 — 누가 만드는 건가요?"* | "**LLM이 자동 생성** 합니다. 위키에 새 노트가 추가되면 LLM이 *기존 위키의 분류 체계* 를 보고 *어디에 속하는지* 태그를 붙임. 사람이 후보정할 수 있지만 기본은 자동. *북키핑 비용을 LLM이 받아낸다* 의 구체적 예." |
| 3A-5 | *"'컴파일된 지식' — 컴파일이 뭔가요?"* | "원래는 *코드를 실행 가능한 형태로 변환* 의 뜻이지만, 여기서는 *raw 자료를 정리·연결된 지식 단위로 변환* 의 비유. Karpathy가 자기 글에서 *'compile a wiki'* 라고 표현한 데서 옴. *raw 자료(보고서·논문) → 정리된 위키 페이지* 변환 과정." |
| 3A-6 | *"비교표의 *raw 재검색·재조립* — raw가 뭔가요?"* | "*가공되지 않은 원본 문서* — 보고서·논문·기사 PDF 그 자체. *raw 재검색* 은 *질문이 올 때마다 그 원본 더미를 처음부터 뒤진다* 는 뜻. 정리된 *위키 페이지* 를 두고도 매번 원본부터 다시 보는 비효율 강조." |

## 메타 — 이 문서 사용법

1. **발표 직전 리뷰용.** 발표자가 모든 질문에 대비할 필요는 없음 — *최소 A·B 카테고리* 는 한 번 훑고 들어갈 것.
2. **슬라이드 본문 Q&A 표와 분리.** 슬라이드 본문(`최종_v1_pageN.md`) 의 Q&A 표는 *3-4개의 핵심* 만 유지. 나머지는 이 문서에서.
3. **신규 질문 발견 시 이 파일에 누적.** 리허설/사전 검토에서 새로운 질문이 나오면 해당 페이지·카테고리에 추가.
4. **답변 톤.** 임원 보고 톤 — *한 호흡에 한 메시지*, 짧게. 길어지면 *그건 다음 슬라이드에서* 로 넘김.
