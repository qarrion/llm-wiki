## 이 페이지의 정체

### 목적 — 무엇을 위한 한 장인가

슬라이드 3에서 *"같은 LLM, 다른 구조"* 다리를 받아 — **그 *구조* 가 정확히 어떻게 생겼는지** 를 한 화면에 풀이하는 슬라이드.
*추상* 으로 가지 않고 **사람이 평소 하던 5단계 정리 흐름 vs LLM이 같은 5단계를 자동으로 도는 흐름** 을 좌·우 평행 비교로 박음.

발표자 입장에서는 *"내가 하던 일을 LLM이 그대로, 단지 *저비용·매일·자동* 으로"* 라는 메시지를 30초 안에 흡수시키는 자리.
청중 입장에서는 *"이게 새로운 시스템이 아니라 내가 익숙한 5단계의 자동화"* 임을 즉시 인지 → *"우리 본부 일에 적용 가능한가"* 의문이 슬라이드 5(본부 효용)로 자연 진입.

### 설명 대상 — 무엇을 설명하는가

이 한 장이 임원의 머리에 박아 넣는 **4가지 정보** :

| # | 정보 | 화면 위치 |
|---|---|---|
| ① | **5단계 평행 흐름** — 자료 흡수 → 해석/추출 → 정리 → 저장 → 꺼내보기 | 중앙 좌·우 5행 (↔ 화살표) |
| ② | **저비용 + 매일 + 자동** — LLM이 같은 일을 *어떤 조건* 으로 하는가 | 헤드라인 우측, 오렌지 강조 |

| ④ | **Claude = 지능** — 추출 / 작성·태그 / 링크 자동 / 모순 발견 | 우측 오렌지 박스 (②⑤ 화살표) |

### 설명 방식 — 어떻게 설명하는가

1. **헤드라인 = 비유 + 조건 한 줄.**
   - *"사람이 하던 정리, LLM이 그대로. 저비용 + 매일 + 자동"* — 본문이 *동일성* 을, 우측 오렌지 *저비용 + 매일 + 자동* 이 *조건* 을 박음. 임원이 한 줄에서 *"같은 일, 다른 조건"* 흡수.
   - 초안 v5의 *"0 비용으로"* → **"저비용 + 매일 + 자동"** 으로 확장. *0 비용* 한 단어보다 *3가지 조건의 합* 이 *왜 LLM이 가능한가* 를 더 분명히 풀이.
2. **좌·우 5단계 평행 = "내가 하던 일" 인식 점등.**
   - 좌측 헤더 *"수기 — 사람이 직접"* (회색) — *기존, 익숙*.
   - 우측 헤더 *"자동 — Claude Code + Obsidian"* (검정 + 오렌지 사이드 바) — *새로움, 무게*.
   - ↔ 양방향 화살표가 *"같은 일이다"* 의 시각화. 5단계 모두 1:1 대응 → *"내가 하던 일이 디지털화"* 로 즉시 인지.
3. **우측 도구 설명 박스 2개 = *어떤 도구가 어디에 작동* 풀이.**
   - **OBSIDIAN 저장소 (보라):** 마크다운 파일 시스템 / [[링크]] 노트 연결 / 그래프 뷰 시각화 / 로컬 저장·검색. 화살표가 ③ `.md 파일 자동 정리` + ④ `Obsidian vault 자동 저장` 를 가리킴.
   - **CLAUDE 지능 (오렌지):** 자료 읽고 핵심 추출 / 노트 작성·태그 부여 / [[링크]] 자동 연결 / 모순·패턴 발견. 화살표가 ② `Claude Code 추출` + ⑤ `query·그래프 뷰` 를 가리킴.
   - **두 도구가 5단계를 *분업* 하고 있음** 이 시각적으로 명확 — 임원이 *"새 시스템"* 으로 받지 않고 *"기존 도구 두 개의 결합"* 으로 받음.
4. **슬라이드 1 *Claude + Obsidian = Knowledge Graph* 수식의 풀이.**
   - 슬라이드 1에서 던진 수식이 여기서 *동작 단위* 로 분해됨 — Claude는 ②⑤(지능 작업), Obsidian은 ③④(저장 작업).
   - ⑤ *나중에 query · 그래프 뷰* 가 슬라이드 1의 *Knowledge Graph* 그림 회수.
5. **초안 v5 → 최종에서 빠진 요소들.**
   - *"핵심 아이디어 :"* prefix — 슬라이드 1·2·3 일관성 따라 삭제.
   - 단계별 부제 *(책·신문·회의 노트)* / *(찾기 어려움 / 휘발 / 정리비용 ↑)* — 박스 라벨에 흡수, 시각 정리.
   - Karpathy 인용 *"Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스"* — 이 슬라이드에서 삭제. 슬라이드 6 헤드라인으로 승격.
   - 마무리 한 줄 *"사람이 평생 한 번 하던 정리를, LLM은 매일 한다"* — 헤드라인 *"저비용 + 매일 + 자동"* 에 흡수.

### 발표자 멘트

> *그러면 클로드와 옵시디언이 *어떤 프로세스* 로 돌아가는지, 사람이 수기로 할 때와 비교해서 보여드리겠습니다.*
>
> *왼쪽이 우리가 평소에 하던 *수기* 정리 흐름입니다. ① 자료를 본다 → ② 머리로 해석한다 → ③ 노트에 적는다 → ④ 폴더에 넣는다 → ⑤ 나중에 꺼내본다. **다섯 단계 모두 사람이 직접 합니다.** 그래서 시간이 지나면 정리비용이 쌓이고, 휘발되고, 찾기 어려워집니다.*
>
> ***그런데 — 제가 생각하는 가장 고통스러운 부분은 ④번입니다.*** *③번에서 개별 노트 하나하나는 한 번에 정리한다고 해도, 그것을 *기존 노트들과의 관계* 안에 위치시키고 내용을 업데이트하는 일은 — **노트가 늘어날수록 비용이 *지수적* 으로 상승합니다.** 그동안 옵시디언 같은 도구가 잘 안 됐던 진짜 이유가 여기 있습니다.*
>
> *반면 오른쪽 — LLM이 같은 일을 하는 흐름입니다. **단계가 똑같습니다.** ① 자료가 들어오면 → ② Claude Code 같은 LLM 에이전트가 읽고 추출 → ③ 마크다운 파일로 정리 → ④ Obsidian vault 에 자동 저장 → ⑤ 나중에 query 또는 그래프 뷰로 꺼내봅니다. 그러나 **다섯 단계 모두 자동으로 돌아갑니다.***
>
> *여기서 잠깐 옵시디언과 클로드의 역할을 보시면 — **옵시디언은 *저장소* 처럼 보일 수 있지만, 사실은 *뷰어(viewer)* 에 가깝습니다.** 마크다운 파일을 보관하고, 링크로 연결하고, 그래프 뷰로 시각화합니다 — ③ ④ 단계. 반대로 **클로드는 *지능* 입니다.** 자료를 읽고 핵심을 추출하고, 노트를 작성하고, 모순·패턴을 발견합니다 — ② ⑤ 단계. **슬라이드 1에서 보여드린 *Claude + Obsidian = Knowledge Graph* 수식이 — 단계 단위로 분업한 게 바로 이 그림입니다.***
>
> *그래서 헤드라인이 — *저비용 + 매일 + 자동.* **사람이 평생 한 번 하던 정리를, LLM은 매일 합니다.** 그것도 별도 인프라 없이, 마크다운 폴더 하나에서.*
>
> *그러면 다음 장에서는 — **클로드가 구체적으로 어떤 작업을 하는지** 알아보겠습니다."*

| *"저비용이 정확히 얼마나?"* | "마크다운 파일 폴더 하나가 시작점입니다. 별도 서버·인프라 없습니다. *비용* 은 LLM API 호출 비용 정도 — Claude/ChatGPT 한 달 구독료 수준에서 시작합니다." |
