# 최종 v1 — 슬라이드 5 (page05.png 역추출)

## 이 페이지의 정체

### 목적 — 무엇을 위한 한 장인가

발표 전체의 **본체(Center of Gravity)** 슬라이드. 슬라이드 4에서 *"클로드가 구체적으로 어떤 작업을 하는지"* 다리를 받아 — **Claude(LLM)가 어떤 데이터 구조 위에서, 어떤 명령으로, 어떻게 wiki를 자라게 하는가** 를 한 화면에서 끝까지 풀이하는 자리.

발표자 입장에서는 *역할·구조·명령·데이터 흐름* 4가지를 한 다이어그램으로 동시에 박는 자리 — *"이 한 그림이 오늘 발표의 본체"* 라고 선언 가능한 슬라이드.
청중 입장에서는 슬라이드 1·3·4 에서 던져 둔 신호들이 **여기서 합쳐져서 작동** 함을 인지 — *Wiki = Codebase*, *LLM = Programmer*, *raw → wiki → output* 흐름이 한 화면에서 동시 회수.

### 설명 대상 — 무엇을 설명하는가

이 한 장이 임원의 머리에 박아 넣는 **5가지 정보** :

| # | 정보 | 화면 위치 |
|---|---|---|
| ① | **변화 메시지** — 일회성에서 *쌓고 진화하는* LLM 으로 | 상단 헤드라인 (두 줄, *쌓고 진화하는* 오렌지 강조) |
| ② | **Karpathy 비유 회수** — Obsidian = IDE / LLM = 프로그래머 / Wiki = 코드베이스 | 헤드라인 우측 인용 박스 (오렌지 사이드 바) |
| ③ | **데이터 파이프라인** — INPUT → raw/ → wiki/ ← schema/ → query → OUTPUT (피드백 루프) | 중앙 본체 다이어그램 (`WIKI = CODEBASE` 레이블) |
| ④ | **LLM 의 3가지 명령** — ingest / query / lint (= 자정화) | 좌하단 별도 박스 (`LLM = PROGRAMMER`) |
| ⑤ | **위키 예시** — 종목·섹터·매크로 + index·log 트리 (본부 폴더 그대로) | 우하단 zoom-in 박스 (*위키 예시*) |

### 설명 방식 — 어떻게 설명하는가

1. **헤드라인 = 변화 메시지 두 줄.**
   - 슬라이드 1·3·4의 prefix(*"문제 정의 :", "핵심 아이디어 :"*) 와 다른 **두 줄 단정형** 으로 무게 이동. *"일회성으로 매번 답을 찾는 LLM에서 / 지식을 스스로 쌓고 진화하는 LLM으로"* — 본문 다이어그램이 *원리* 를 풀이하므로, 헤드라인은 *"이게 어떤 변화인가"* 라는 *메시지* 를 박음.
   - 우상단 *"SLIDE 05 · 핵심 아이디어"* 는 페이지네이션 신호 (이 페이지가 *핵심 아이디어* 자리임을 임원에게 명시).
2. **헤드라인 옆 Karpathy 인용 = 비유 권위 회수.**
   - *"Obsidian은 IDE, LLM은 프로그래머, Wiki는 코드베이스다."* — 슬라이드 1에서 *Claude + Obsidian = Knowledge Graph* 수식으로 던졌고, 슬라이드 4에서 *Obsidian = 뷰어 / Claude = 지능* 으로 풀었던 비유가, 여기서 **Karpathy 원문 형태로 단정 회수**. 이 인용 한 줄로 다이어그램 전체에 *코딩 패러다임* 의 무게가 입혀짐.
3. **중앙 다이어그램 = `WIKI = CODEBASE` 박스 안에 raw / wiki / schema 3구조.**
   - **`WIKI = CODEBASE`** 레이블이 박스 상단에 박혀 있어 *"이 박스 전체가 코드베이스"* 임을 즉시 인지.
   - 좌측 `raw/` (원본·감사·역추적용 / .docx, .pptx, .pdf, mp4, .md) — *손대지 않는 원본*.
   - 중앙 `wiki/` (검정 배경, 강조 — *정리된 지식 / 답변의 본체 / 주제별 요약.md / Index.md / Log.md*) — *본체*.
   - 우측 `schema/` (본부 톤·강조 기준 / CLAUDE.md, Rules.md, Memory.md) — *운영 규칙 — 사람이 한 번 박고 LLM이 따른다*.
   - 화살표 라벨: *정리* (raw → wiki, ingest), *주입* (schema → wiki), *질의* (wiki → output, query), *점검* (lint, wiki 자체로 돌아가는 화살표).
   - **상단 점선 피드백 루프 *"답변·문서가 다시 자료가 된다"*** — *"답변이 휘발이 아니라 다시 자료로"* 메시지를 점선 한 줄로 시각화. 슬라이드 3 비교표 ②행 *"좋은 답변은 위키 페이지로 적립"* 의 직접 회수.
4. **좌하단 `LLM = PROGRAMMER` 박스 = 명령 셋 풀이.**
   - 슬라이드 4의 *"다음 장에서는 클로드가 구체적으로 어떤 작업을 하는지"* 다리에 대한 직접 응답.
   - **ingest** — 외부 자료 → raw → 정리하여 wiki에 반영
   - **query** — wiki를 읽어 답변·문서 생성 (좋은 답은 다시 위키로)
   - **lint** — wiki 주기적 자체 점검 (모순·누락·오래됨)
   - 다이어그램의 화살표 라벨이 명령 셋의 시각화이고, 이 박스가 명령 셋의 풀이 — **다이어그램 + 박스가 *그림 + 사전* 관계.**
5. **우하단 `위키 예시` zoom-in 박스 = 추상 + 본부 구체 동시.**
   - 좌측 첫 줄 *"주제별로 요약 .md가 만들어진다"* = **추상 형태**.
   - 그 아래 트리 = **본부 적용 예시** (종목/삼성전자.md, 섹터/메모리.md, 매크로/USDKRW.md, index.md, log.md).
   - 한 박스에서 *"왜 이렇게 정리하나"* + *"내 일에 어떻게 들어오나"* 가 동시에 답해짐 → 임원이 *"우리 본부에 적용 가능한가"* 의문을 슬라이드 위에서 즉시 해소.
6. **본부 효용 라벨 = INPUT·OUTPUT 한국어로만 압축.**
   - 좌측 INPUT 아래 *"자료 / 모든·메모"*, 우측 OUTPUT 아래 *"답변 / 답변·보고서"* — 영어 라벨 + 본부 한국어 부속 한 줄 형태로 가벼움.
   - 초안 v5의 *"셀사이드 PDF · 사내 메모 · IR 자료"* 같은 긴 부속 라벨은 압축 — `위키 예시` 박스가 *내가 본부 일에 어떻게 들어오나* 를 이미 답하므로 INPUT·OUTPUT 라벨은 가볍게.

### 발표 후 청중 머릿속에 남길 한 줄

> *"raw → wiki → output 한 줄 흐름. WIKI가 코드베이스고, LLM이 ingest·query·lint 세 명령으로 그 위에서 작동한다. 위키 예시 박스를 보니 본부 분류 그대로네."*

- **`raw/`** (좌측 박스): 원본·감사·역추적용. 파일 형식 명시 (.docx, .pptx, .pdf, mp4, .md).

- **화살표 라벨 (오렌지):** *정리(ingest)* / *주입* / *질의(query)* / *점검(lint)*.
- **상단 점선 피드백 루프:** *"답변·문서가 다시 자료가 된다"* — output 이 다시 input 으로.


각 명령 라벨(ingest / query / lint) 은 오렌지 박스로 시각 강조 — 다이어그램 화살표 라벨과 동일 색·동일 단어로 일치.

### 발표자 멘트

> *"앞 장에서 클로드와 옵시디언이 5단계 자동화를 한다고 보여드렸습니다. 그 자동화가 *구체적으로 어떤 데이터 구조 위에서, 어떤 명령으로* 작동하는지 — 이 한 그림이 오늘 발표의 본체입니다.*
>
> ***좌→우 한 줄로 보시면 됩니다.** 왼쪽 INPUT — 본부에 매일 들어오는 자료 (셀사이드 PDF·사내 메모·IR·매크로 노트). 가운데 박스가 **WIKI = CODEBASE** — 카파시 표현으로 *코드베이스* 입니다. 오른쪽 OUTPUT — 운용역이 받는 답변, 보고서.*
>
> ***WIKI = CODEBASE 박스 안을 들여다보면 세 종류의 데이터가 있습니다.** **`raw/`** 는 원본 그대로 보관 — 감사·역추적용. **`wiki/`** 는 정리된 지식 — *답변의 본체* 입니다. **`schema/`** 는 운영 규칙 — *본부 톤, 강조 기준, 컴플라이언스* 를 사람이 한 번 박아두면 LLM이 따릅니다. CLAUDE.md, Rules.md, Memory.md.*
>
> ***데이터를 움직이는 건 LLM 입니다 — 카파시 비유로 *프로그래머*.** 좌하단 박스를 보시면 LLM이 하는 일이 세 가지입니다. **`ingest`** — 자료가 들어오면 정리해서 wiki에 반영. **`query`** — 질문에 답할 때 wiki를 읽고 출력, 좋은 답은 다시 위키로. **`lint`** — 주기적으로 wiki 자체를 점검 (모순·누락·오래된 내용). **이게 *자정화* 입니다.***
>
> ***그리고 — 위 점선 화살표 보이시죠.*** *"답변·문서가 다시 자료가 된다"* — **답이 채팅에 휘발이 아니라 다시 위키로 들어옵니다.** 슬라이드 3에서 보신 *비교표 두 번째 행* 이 바로 여기서 작동합니다.*
>
> ***우하단 *위키 예시* 박스를 보시면 — wiki/ 안이 어떻게 생겼는지 한 단계 더 펼친 그림입니다.*** *주제별로 요약 .md가 만들어집니다* — 종목별로 *삼성전자.md*, 섹터별로 *메모리.md*, 매크로별로 *USDKRW.md*. 그리고 *index.md* 가 전체 색인을 LLM이 자동 유지, *log.md* 가 작업 로그·감사 흔적입니다. **사람이 본부 폴더 구조를 짜듯 LLM도 그 구조 위에서 정리합니다.***
>
> *그러면 — *누가 이 구조를 시작하고, 누가 운영하는가* — 마지막 장에서 보여드리겠습니다."*

| *"raw, wiki, schema 차이가 뭔가?"* | "*raw* 는 손대지 않은 원본 (감사·역추적용). *wiki* 는 LLM이 정리한 지식 (종목·섹터 페이지). *schema* 는 사람이 박아둔 운영 규칙 (`CLAUDE.md`, `Rules.md`, `Memory.md`). 셋이 분리되어야 *원본 추적* 과 *정리된 답* 이 모두 가능합니다." |
| *"ingest/query/lint 가 다 자동인가? 사람은 안 끼나?"* | "기본은 자동. 다만 *Lint* 가 모순을 표시하면 *어느 쪽이 맞는지* 판단은 사람. *Query* 답변도 *위키로 적립* 단계에서 사람 검토 모드를 둘 수 있습니다." |
| *"schema/ 안의 CLAUDE.md / Rules.md / Memory.md 차이가 뭔가?"* | "**CLAUDE.md** 는 *본부 톤·강조 기준* 의 메인 규칙 (한 페이지). **Rules.md** 는 세부 규칙 (예: *컴플라이언스 한 줄*). **Memory.md** 는 LLM이 누적해서 학습할 본부 고유 맥락. 한 번 박으면 LLM이 모든 .md 작성에 따릅니다." |
| *"wiki/ 폴더 분류(종목/섹터/매크로)는 누가 정하나?"* | "**처음 한 번 사람이 정합니다** — 본부 운용 분류 그대로. 새 종목·섹터가 추가되면 LLM이 schema 규칙에 따라 적절한 폴더에 자동 배치. 분류 자체를 바꾸려면 사람이 schema 한 줄 수정." |
| *"index.md, log.md는 누가 만드나?"* | "**LLM이 자동 유지** 합니다. 새 .md가 추가되면 색인이 갱신되고, 모든 작업이 log에 기록 — 감사·역추적용. 사람이 손댈 일 없습니다." |
| *"점선 피드백 루프 (답변이 자료가 된다) 가 정확히 어떻게 작동하나?"* | "운용역 query 에 대한 *좋은 답* 이 사람 검토 후 wiki 페이지로 다시 들어갑니다. 그래서 다음 사람의 query는 그 답까지 포함된 더 풍부한 wiki 위에서 작동합니다. 슬라이드 3 비교표 ②행이 바로 이 루프." |
| *"기존 RAG 시스템 도입과 뭐가 다른가?"* | "RAG는 *질문할 때마다 raw에서 chunk를 가져옴* — 정리는 안 됨. LLM-WIKI는 *정리된 wiki를 직접 읽음* — 답변 품질 + 추적 가능성 모두 다릅니다. 슬라이드 3 비교표 ②③행이 그 차이." |
| *"위키가 *판단* 도 하나? (종목 추천 같은)"* | "**아닙니다.** 위키는 정리·관계까지 — 결론(*'사야 한다'*)은 만들지 않습니다. *Lint* 도 모순 *표시* 까지, *어느 쪽이 맞는지* 는 사람. 슬라이드 6에서 그 선이 한 번 더 명시됩니다." |
| *"문서가 100개 넘으면 LLM이 다 못 읽지 않나?"* | "카파시 본인이 *~100개 문서·40만 단어* 까지 별도 인프라 없이 작동한다고 명시. 본부 첫 파일럿 규모는 이 안에 들어옵니다 — `index.md` (색인) 를 LLM이 자동 유지." |

### 슬라이드 6과의 연결

**다리 한 줄 (발표자 멘트 마지막):**

> *"그러면 — *누가 이 구조를 시작하고, 누가 운영하는가* — 마지막 장에서 보여드리겠습니다."*

→ 슬라이드 6 헤드라인으로 자동 진입. 다이어그램의 *데이터 흐름* 과 *명령 셋* 이 슬라이드 6에서 **PM(우리)이 하는 일** 로 풀이됨.

청중 머리 속 흐름:
1. *(슬라이드 4)* "사람 5단계 vs LLM 5단계. Obsidian = 뷰어, Claude = 지능. 근데 그 *지능* 이 구체적으로 뭘 하는데?"
2. *(슬라이드 5 헤드라인)* "일회성에서 *쌓고 진화* 로. 변화 메시지 명확."
3. *(Karpathy 인용 회수)* "Obsidian = IDE, LLM = 프로그래머, Wiki = 코드베이스. 이제 슬라이드 1 수식의 본격 풀이."
4. *(다이어그램 좌→우)* "INPUT → raw → wiki → schema 주입 → query → OUTPUT. 한 줄 흐름."
5. *(`WIKI = CODEBASE` 박스)* "raw·wiki·schema 3구조. 원본·정리·규칙 분리."
6. *(피드백 루프 점선)* "답변이 다시 자료로. 슬라이드 3 비교표가 여기서 작동."
7. *(좌하단 LLM = PROGRAMMER 박스)* "ingest·query·lint 세 명령. 슬라이드 4의 *Claude 지능* 이 이렇게 구체화."
8. *(우하단 위키 예시)* "wiki/ 안에 종목/섹터/매크로 + index/log. **본부 분류 그대로네.**"
9. *(자동 의문)* "그래서 누가 시작하고 누가 운영?"
10. *(슬라이드 6)* "각자가 PM. 폴더 하나에서 오늘부터."
